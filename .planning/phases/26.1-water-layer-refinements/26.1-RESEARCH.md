## Summary

Phase 26.1 is a refinement pass on the Phase 26 water stress layer. The work spans five domains: (1) tiered Overpass queries splitting priority countries (IL, JO, LB, SY, IQ, IR, AF) from non-priority countries with different notability thresholds, (2) expanding the stress scoring scale to 0-10 with a "destroyed" state triggered by destructive CAMEO events, (3) auditing desalination plant coverage against known major Gulf facilities, (4) creating 3-4 new 32x32 canvas icon markers for facility types (dam, reservoir, treatment_plant, desalination), and (5) adding a Vercel cron job to refresh water cache daily.

The existing codebase is well-structured for these changes. The Overpass adapter (`server/adapters/overpass-water.ts`) already uses per-type bbox queries with geographic exclusion, making the tiered priority/non-priority split a natural extension. The icon atlas (`src/components/map/layers/icons.ts`) uses a canvas-based mask-mode approach that scales linearly by expanding canvas width. The stress scoring module (`src/lib/waterStress.ts`) is self-contained with clear conversion functions. The Vercel cron pattern is already established with `cron-health.ts`.

**Primary recommendation:** Implement in 4-5 waves: (1) server-side tiered queries + treatment_plant type, (2) stress scoring 0-10 with destroyed state, (3) canvas icon atlas expansion, (4) Vercel cron setup, (5) desalination audit + test verification.

## Architecture Patterns

### Tiered Overpass Query Strategy

**Decision:** Use single bbox queries per facility type (current pattern), then post-filter results by country tier.

**Rationale:**

- Current `ME_BBOX = '15,30,42,70'` covers all target countries
- Overpass rate limits are per-connection, not per-query-size
- Splitting into per-country queries would multiply from 3 queries (dams, reservoirs, desalination) to potentially 3 x 2 tiers = 6+ queries
- Post-filtering is deterministic and testable
- Treatment plant queries ONLY run for priority countries using an additional bbox-filtered or country-filtered query

**Implementation approach:**

1. Keep existing 3 queries (dams, reservoirs, desalination) on full ME_BBOX -- these return all named facilities
2. Add 1 new query for treatment plants: `nwr["man_made"="water_works"]["name"]` on full ME_BBOX
3. Post-filter: for non-priority countries, only keep dams/reservoirs that have Wikipedia or Wikidata tags (i.e., `tags.wikidata` or `tags.wikipedia` exists)
4. Post-filter: for non-priority countries, DROP all treatment plants entirely
5. Desalination passes through unfiltered for all countries (per user decision)

**Priority country detection:** Use a reverse-geocoding approximation -- classify by nearest country centroid from the existing `CENTROIDS` + `COUNTRY_CENTROIDS` data. The `basinLookup.ts` already has 29 country centroids with [lat, lng]. Reuse those for country classification.

**Priority country ISO mapping to centroid names:**

- IL -> Israel
- JO -> Jordan
- LB -> Lebanon
- SY -> Syria
- IQ -> Iraq
- IR -> Iran
- AF -> Afghanistan

### Notability Filter for Non-Priority Countries

**What constitutes "Wikipedia/Wikidata-tagged":** In OSM, notable features have tags like:

- `wikidata=Q1234567` (Wikidata item ID)
- `wikipedia=en:Article_Name` (Wikipedia article link)

**Overpass query approach:** Rather than querying with the tag filter in Overpass (which would require separate queries), fetch all named facilities and post-filter:

```typescript
function isNotable(tags: Record<string, string>): boolean {
  return !!(
    tags.wikidata ||
    tags.wikipedia ||
    Object.keys(tags).some((k) => k.startsWith('wikipedia:'))
  );
}
```

This keeps the Overpass query count at 4 (dams, reservoirs, desalination, treatment_plants) and applies tiering logic in the normalization step.

### Score 0 (Destroyed) State

**Architecture:** The "destroyed" detection runs client-side using the existing event store, not server-side. This matches the existing `attackStatus.ts` pattern.

**Where to compute:** In `useWaterLayers.ts` when generating icon colors, and in `WaterFacilityDetail.tsx` for display. The composite health score from the server stays unmodified (0-1 based on WRI + precipitation). Score 0 is a client-side override triggered by destructive events.

**Alternative considered:** Computing destroyed state server-side. Rejected because:

- Server doesn't have access to the event store
- Client already has all events loaded
- `attackStatus.ts` already does this exact pattern for sites
- Keeping it client-side means the destroyed state responds to event data changes in real time

### Destructive CAMEO Codes for Score 0

**Recommendation:** The following CAMEO base codes should qualify as "destructive" for triggering score 0:

| CAMEO Code | Description            | ConflictEventType | Rationale                         |
| ---------- | ---------------------- | ----------------- | --------------------------------- |
| 183        | Bombing                | `bombing`         | Direct infrastructure destruction |
| 194        | Artillery/tank support | `shelling`        | Direct infrastructure destruction |
| 195        | Aerial weapons         | `airstrike`       | Direct infrastructure destruction |
| 204        | WMD                    | `wmd`             | Catastrophic destruction          |

**Implementation:** Filter by ConflictEventType rather than CAMEO codes, since the type classification is already done. Destructive types: `airstrike`, `bombing`, `shelling`, `wmd`.

```typescript
const DESTRUCTIVE_TYPES = new Set(['airstrike', 'bombing', 'shelling', 'wmd']);

function isDestroyed(facility: WaterFacility, events: ConflictEventEntity[]): boolean {
  return events.some(
    (e) =>
      DESTRUCTIVE_TYPES.has(e.type) &&
      haversineDistanceKm(facility.lat, facility.lng, e.lat, e.lng) <= 5,
  );
}
```

This is cleaner than CAMEO code filtering because the type system already handles the classification.

### Vercel Cron Configuration

**Current state:** One cron job exists (`/api/cron/health` at midnight daily).

**New cron:** Add `/api/water?refresh=true` on a daily schedule.

**Hobby plan limitation:** Vercel Hobby plan allows **only 1 cron job per day** (crons can only run once per day). Since there's already a health cron, adding a second daily cron is allowed (Hobby supports up to 2 cron jobs total per project, each running once daily max).

**Security:** The existing health cron has no CRON_SECRET check. For consistency, the water cron doesn't need one either (the endpoint already exists and is rate-limited). However, the `refresh=true` parameter should be guarded to prevent external abuse. Options:

- Check for `vercel-cron/1.0` user agent (sent by Vercel automatically)
- Or add CRON_SECRET check

**Recommendation:** Check user agent for cron requests to the refresh endpoint:

```typescript
const isCron = req.headers['user-agent']?.includes('vercel-cron');
const forceRefresh = req.query.refresh === 'true' && isCron;
```

**Schedule:** `0 6 * * *` (06:00 UTC daily) -- offset from health check at midnight to avoid concurrent execution.

**vercel.json addition:**

```json
{
  "crons": [
    { "path": "/api/cron/health", "schedule": "0 0 * * *" },
    { "path": "/api/water?refresh=true", "schedule": "0 6 * * *" }
  ]
}
```

**Route timeout removal:** Per user decision, remove the `Promise.race` 180s timeout from the water route. The cron handles refresh timing. Client requests that hit a cache miss will attempt the fetch without a route-level timeout, but the Vercel function timeout (60s maxDuration) provides the hard cap. If the Overpass fetch takes longer than 60s on Vercel, the function dies and the client gets stale cache (or empty array with stale:true). This is acceptable because:

1. The cron job refreshes daily so the cache should always be warm
2. Client requests to a warm cache return immediately
3. The 90s Overpass timeout is only relevant in local dev (where there's no Vercel function limit)

## Common Pitfalls

### Pitfall 1: Overpass Rate Limiting on Multiple Queries

**What goes wrong:** Adding a 4th query (treatment_plants) increases total Overpass API calls. Running sequentially with the current per-type pattern, each query takes 5-10s. Four queries = 20-40s, which fits within the 60s Vercel function limit but is tight.
**Why it happens:** Overpass public API rate limits to ~2 concurrent connections per IP.
**How to avoid:** Keep queries sequential (current pattern), add 1s delay between queries, keep the 90s per-query timeout. The cron-triggered refresh can afford longer execution. For client requests, the cache should always be warm.
**Warning signs:** 429 responses from Overpass, timeout on 3rd/4th query.

### Pitfall 2: Stale Test Assertions for `classifyWaterType`

**What goes wrong:** The existing test file `server/__tests__/adapters/overpass-water.test.ts` has tests for `treatment_plant` and `canal` types that currently fail (or were skipped) because those types were removed in Phase 26. Adding `treatment_plant` back will fix some tests but `canal` remains removed.
**Why it happens:** Test file was written for the original design, production code was trimmed.
**How to avoid:** Update tests to remove `canal` expectations, verify `treatment_plant` assertions are correct, ensure `classifyWaterType` returns `treatment_plant` for `man_made=water_works` tags.
**Warning signs:** Test failures in overpass-water.test.ts after type changes.

### Pitfall 3: Score 0 (Destroyed) Overriding Actual Health Data

**What goes wrong:** A facility gets marked destroyed even from old/low-confidence events, permanently blacking it out.
**Why it happens:** Current `computeAttackStatus` uses all events from the store without date filtering for the lower bound.
**How to avoid:** Use the same dateEnd filter as the current attack status (events must be before dateEnd). Consider also filtering to only recent destructive events (e.g., within last 7 days) to avoid permanent blackout from old events. However, per the existing pattern in `attackStatus.ts`, "once hit it stays hit" until dateEnd, so this is consistent.
**Warning signs:** Too many facilities showing as destroyed due to distant historical events.

### Pitfall 4: Canvas Atlas Width Mismatch

**What goes wrong:** After adding new icons, the HMR guard check `atlas.width === 416` prevents regeneration with the new 544px width, showing stale/missing icons.
**Why it happens:** The guard exists to invalidate old cached atlases during hot module replacement.
**How to avoid:** Update the guard to `atlas.width === 544` when adding new icons. This is already commented in the codebase.
**Warning signs:** Missing/blank icons for water facilities in dev mode.

### Pitfall 5: Vercel Hobby Plan Cron Limits

**What goes wrong:** Adding a second cron job on Hobby plan might exceed limits or cron jobs might not execute at the expected time.
**Why it happens:** Hobby plan cron jobs can run once per day max, and Vercel may invoke them at any point within the specified hour.
**How to avoid:** Accept that the water refresh may run anytime between 06:00-06:59 UTC. This is fine for 24h cached data. Verify after deployment that both crons appear in the Vercel dashboard.
**Warning signs:** Cron not appearing in Vercel Settings > Cron Jobs after deployment.

### Pitfall 6: Treatment Plant Tag Ambiguity

**What goes wrong:** The OSM tag `man_made=water_works` can refer to various water infrastructure, not just treatment plants specifically. Some might be pumping stations or other facilities.
**Why it happens:** OSM tagging is community-driven and not always precise.
**How to avoid:** Accept this imprecision -- the `["name"]` tag requirement already filters to notable facilities. Treatment plants are only included for priority countries where every named water facility is relevant.
**Warning signs:** Unexpected facility types showing up as "Treatment Plant" in the detail panel.

### Tiered Post-Filter Pattern

```typescript
// Source: Architecture recommendation based on existing overpass-water.ts pattern

const PRIORITY_COUNTRIES = new Set([
  'Israel',
  'Jordan',
  'Lebanon',
  'Syria',
  'Iraq',
  'Iran',
  'Afghanistan',
]);

function isPriorityCountry(lat: number, lng: number): boolean {
  let minDist = Infinity;
  let nearest = '';
  for (const [country, [clat, clng]] of Object.entries(COUNTRY_CENTROIDS)) {
    const d = haversineKm(lat, lng, clat, clng);
    if (d < minDist) {
      minDist = d;
      nearest = country;
    }
  }
  return PRIORITY_COUNTRIES.has(nearest);
}

}
```

### Score 0 Destroyed Detection

```typescript
// Source: Based on existing attackStatus.ts pattern

const DESTRUCTIVE_EVENT_TYPES = new Set(['airstrike', 'bombing', 'shelling', 'wmd']);

function computeDestroyedSet(
  facilities: WaterFacility[],
  events: ConflictEventEntity[],
  dateEnd: number,
): Set<string> {
  const destroyed = new Set<string>();
  const destructiveEvents = events.filter(
    (e) => DESTRUCTIVE_EVENT_TYPES.has(e.type) && e.timestamp <= dateEnd,
  );

  for (const f of facilities) {
    for (const e of destructiveEvents) {
      // Coarse pre-filter
      if (Math.abs(e.lat - f.lat) > 0.05 || Math.abs(e.lng - f.lng) > 0.05) continue;
      if (haversineDistanceKm(f.lat, f.lng, e.lat, e.lng) <= 5) {
        destroyed.add(f.id);
        break;
      }
    }
  }

  return destroyed;
}
```
