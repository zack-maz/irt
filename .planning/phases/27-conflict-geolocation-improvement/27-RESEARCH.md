# Phase 27: Conflict Geolocation Improvement (GDELT Redo) - Research

**Researched:** 2026-04-09
**Domain:** LLM-based event extraction pipeline, geocoding, event ontology
**Confidence:** MEDIUM (model availability verified, Vercel constraint needs design pivot)

## Summary

Phase 27 rearchitects the GDELT conflict event pipeline by inserting an LLM extraction layer that replaces CAMEO-based classification and ActionGeo-based geolocation with structured extraction from grouped multi-row event bundles. The LLM synthesizes one canonical record per real-world event with precise geocoded location, a simplified 5-type ontology, casualty data, and situation summaries.

The research surfaced three critical findings that affect planning: (1) Cerebras has deprecated `llama-3.3-70b` in favor of `gpt-oss-120b`, which is actually a better model -- available on both providers with strict JSON Schema enforcement, (2) the Vercel Hobby plan limits cron jobs to once per day, making the "every 15 minutes" cron approach from D-03 infeasible as a true Vercel cron -- the LLM processing must instead be triggered lazily on `/api/events` requests with a cooldown, and (3) token budgets are tight but workable: Cerebras free tier provides 1M TPD at 30 RPM, Groq provides 200K TPD at 30 RPM for gpt-oss-120b.

**Primary recommendation:** Use `gpt-oss-120b` on both providers (Cerebras primary, Groq fallback). Trigger LLM processing lazily on `/api/events` cache miss with a 15-minute cooldown (same pattern as backfill). Store LLM-enriched events in a separate Redis key (`events:llm`) alongside raw GDELT (`events:gdelt`).

## CRITICAL: Model & Platform Constraint Updates

Research uncovered three facts that conflict with locked decisions. The planner MUST account for these:

### 1. llama-3.3-70b is DEPRECATED on Cerebras (affects D-02)

**Finding:** Cerebras deprecated `llama-3.3-70b` on 2026-02-16, recommending migration to `gpt-oss-120b`. The model is no longer listed in their current supported models.

**Impact:** D-02 specified "Llama 3.3 70B on both providers." This is no longer available on the primary provider.

**Resolution:** Use `gpt-oss-120b` on both Cerebras and Groq. This is a strictly better model: 120B parameters (MoE, 5.1B active), supports strict JSON Schema mode on both providers, 131K context, and benchmarks above Llama 3.3 70B. Available on Cerebras free tier (1M TPD) and Groq free tier (200K TPD). The spirit of D-02 (highest quality for structured extraction) is better served by gpt-oss-120b.

**Confidence:** HIGH -- verified via Cerebras deprecation page and Groq model docs.

### 2. Vercel Hobby Plan Limits Cron to Once Per Day (affects D-03)

**Finding:** The project is on the Vercel Hobby plan (confirmed via ADR-0002). Hobby plan cron jobs are limited to once per day. The existing crons (`/api/cron/health` daily, `/api/cron/warm` every 12h) already push this -- the 12h warm cron may be silently downgraded to daily by Vercel.

**Impact:** D-03 specified "Vercel cron background job every 15 minutes." This is impossible on Hobby plan.

**Resolution:** Two approaches, recommend the first:

**Option A (Recommended): Lazy on-request processing with cooldown.** The LLM processing triggers on `/api/events` cache miss or stale, gated by a 15-minute cooldown key in Redis (`events:llm-process-ts`). This follows the exact same pattern as the existing `backfillEvents()` + `events:backfill-ts` cooldown. The LLM work happens within the request but the result is cached, so subsequent requests serve from cache. With Vercel Hobby's 60s maxDuration and gpt-oss-120b's ~500-3000 tokens/sec speed, a single LLM call completes in under 5 seconds.

**Option B: External cron trigger.** Use a free external cron service (cron-job.org, EasyCron) to hit `/api/cron/llm-process` every 15 minutes. This keeps LLM out of the request path but adds an external dependency.

**Confidence:** HIGH -- verified via Vercel Hobby plan docs and ADR-0002.

### 3. Groq Free Tier TPD is 200K for gpt-oss-120b, Not 500K (affects D-01)

**Finding:** Groq free tier for `openai/gpt-oss-120b`: 200K TPD, 8K TPM, 30 RPM, 1K RPD. Cerebras free tier: 1M TPD, 64K TPM, 30 RPM.

**Impact:** Combined budget is 1.2M TPD (not 1.5M as estimated). Still workable.

**Token math:** Each GDELT event group (~5-15 rows, ~200 tokens of context per row) plus system prompt (~500 tokens) and output (~300 tokens) = ~2,000-4,000 tokens per LLM call. At 1M TPD (Cerebras alone), that's 250-500 events per day. GDELT produces ~50-200 Middle East conflict events per 15-minute window after filtering, but many are duplicates of the same real-world event. After grouping, expect 10-30 unique events per window. At 96 windows/day = 960-2880 groups. At ~3K tokens per call = 2.9-8.6M tokens/day -- THIS EXCEEDS THE FREE TIER.

**Resolution:** Batch multiple event groups into a single LLM call. Instead of one LLM call per event group, batch 5-10 groups into one call. This reduces calls from ~100/day to ~10-20/day and tokens to ~300K-600K/day, well within the 1M Cerebras budget. The LLM processes a batch of event groups and returns a JSON array of enriched events.

**Confidence:** HIGH -- verified via both provider rate limit docs.

### Pattern 1: Multi-Provider LLM Adapter

**What:** Single adapter that tries Cerebras, falls back to Groq, falls back to null (raw GDELT mode).
**When to use:** Every LLM call in the pipeline.

```typescript
// server/adapters/llm-provider.ts
import OpenAI from 'openai';

const MODEL_ID = 'gpt-oss-120b'; // or 'openai/gpt-oss-120b' for Groq

const cerebras = new OpenAI({
  apiKey: process.env.CEREBRAS_API_KEY,
  baseURL: 'https://api.cerebras.ai/v1',
});

const groq = new OpenAI({
  apiKey: process.env.GROQ_API_KEY,
  baseURL: 'https://api.groq.com/openai/v1',
});

export async function callLLM(
  messages: OpenAI.Chat.ChatCompletionMessageParam[],
  schema: Record<string, unknown>,
): Promise<string | null> {
  const params = {
    model: MODEL_ID,
    messages,
    response_format: {
      type: 'json_schema' as const,
      json_schema: { name: 'event_extraction', strict: true, schema },
    },
  };

  // Try Cerebras first
  try {
    const res = await cerebras.chat.completions.create(params);
    return res.choices[0]?.message?.content ?? null;
  } catch (err) {
    log.warn({ err, provider: 'cerebras' }, 'LLM call failed, trying Groq');
  }

  // Fallback to Groq
  try {
    const groqParams = { ...params, model: `openai/${MODEL_ID}` };
    const res = await groq.chat.completions.create(groqParams);
    return res.choices[0]?.message?.content ?? null;
  } catch (err) {
    log.warn({ err, provider: 'groq' }, 'LLM fallback failed');
  }

  return null; // Both failed -- caller uses raw GDELT mode
}
```

### Pattern 3: Lazy LLM Processing with Cooldown

**What:** Process events through LLM on `/api/events` cache miss, with a 15-minute cooldown.
**When to use:** Instead of a Vercel cron (which can't run every 15 min on Hobby).

```typescript
// In server/routes/events.ts (extended pattern)
const LLM_PROCESS_KEY = 'events:llm-process-ts';
const LLM_COOLDOWN_MS = 900_000; // 15 minutes

async function shouldRunLLM(): Promise<boolean> {
  try {
    const lastTs = await redis.get<number>(LLM_PROCESS_KEY);
    if (!lastTs) return true;
    return Date.now() - lastTs > LLM_COOLDOWN_MS;
  } catch {
    return true;
  }
}

// In the route handler, after fetching raw GDELT:
if (await shouldRunLLM()) {
  const groups = groupGdeltRows(rawEvents);
  const enriched = await processGroupsWithLLM(groups); // Returns null on total LLM failure
  if (enriched) {
    await cacheSetSafe('events:llm', enriched, LLM_REDIS_TTL_SEC);
    await redis.set(LLM_PROCESS_KEY, Date.now(), { ex: 3600 });
  }
}

// Serve LLM-enriched events if available, else raw GDELT
const llmCache = await cacheGetSafe<EnrichedEvent[]>('events:llm', LLM_LOGICAL_TTL);
const events = llmCache?.data ?? rawGdeltEvents;
```

### Pattern 4: Nominatim Forward Geocoding

**What:** Extend existing `nominatim.ts` with a forward search endpoint.
**When to use:** After LLM extracts a place name, resolve to lat/lng.

```typescript
// server/adapters/nominatim.ts (add to existing file)
export async function forwardGeocode(
  placeName: string,
  countryCode?: string,
): Promise<{ lat: number; lng: number; displayName: string; type: string } | null> {
  const params = new URLSearchParams({
    q: placeName,
    format: 'jsonv2',
    limit: '1',
    'accept-language': 'en',
  });
  if (countryCode) params.set('countrycodes', countryCode);

  const url = `https://nominatim.openstreetmap.org/search?${params}`;
  const res = await fetch(url, {
    headers: { 'User-Agent': 'IranConflictMonitor/1.0 (personal project)' },
  });
  if (!res.ok) return null;
  const data = await res.json();
  if (!data[0]) return null;
  return {
    lat: parseFloat(data[0].lat),
    lng: parseFloat(data[0].lon),
    displayName: data[0].display_name,
    type: data[0].type,
  };
}
```

### Pattern 5: Batch LLM Calls for Token Efficiency

**What:** Send multiple event groups in a single LLM call.
**When to use:** Always -- single-event calls would exceed token budget.

```typescript
// Batch 5-10 event groups per LLM call
const BATCH_SIZE = 8;
const batches = chunk(eventGroups, BATCH_SIZE);
const results: EnrichedEvent[] = [];
for (const batch of batches) {
  const content = await callLLM(buildBatchPrompt(batch), batchSchema);
  if (content) {
    const parsed = JSON.parse(content);
    results.push(...parsed.events);
  }
}
```

### Anti-Patterns to Avoid

- **Single LLM call per GDELT row:** Token budget busting. Always group + batch.
- **LLM in the hot request path without caching:** Every client poll would trigger LLM. Use cooldown + cache.
- **Trusting LLM output without Zod validation:** LLM can hallucinate invalid types or coordinates. Always validate with Zod after JSON parse.
- **Hard dependency on LLM availability:** The map must never go blank. Always have the raw GDELT fallback path.
- **Blocking on Nominatim geocoding in the LLM response loop:** Nominatim has a 1 req/sec policy. Batch geocoding separately, use cache aggressively.

## Common Pitfalls

### Pitfall 1: Token Budget Exhaustion

**What goes wrong:** Each GDELT poll produces 50-200 filtered rows. If each row triggers its own LLM call at ~3K tokens, daily usage hits 5-10M tokens -- 5-10x the free tier.
**Why it happens:** Naive 1:1 row-to-LLM mapping without grouping or batching.
**How to avoid:** Group rows into real-world events (10-30 groups per poll), then batch 5-10 groups per LLM call. Target 10-20 LLM calls per processing cycle, ~3-5K tokens per call = ~50-100K tokens per cycle. At 96 cycles/day = 4.8-9.6M -- still too much! Must process only NEW events (diff against cached), not all events every cycle.
**Warning signs:** 429 rate limit errors from Cerebras/Groq within hours of deployment.

### Pitfall 2: Nominatim Rate Limiting

**What goes wrong:** Nominatim has a strict "1 request per second" usage policy. Batch-geocoding 30 LLM-extracted place names in parallel triggers a ban.
**Why it happens:** Ignoring Nominatim's usage policy.
**How to avoid:** Sequential geocoding with 1-second delays between calls. Aggressive Redis caching (existing 30-day TTL). Coordinate quantization (existing 2-decimal-place pattern). Many events will hit the same cities repeatedly -- cache hit rate should be high.
**Warning signs:** 403/429 from Nominatim, or your User-Agent getting blocked.

### Pitfall 3: LLM Hallucinating Locations

**What goes wrong:** LLM extracts "near Isfahan" and confidently outputs coordinates that are 200km from Isfahan.
**Why it happens:** LLM training data contains approximate geographic knowledge, not precise coordinates.
**How to avoid:** LLM outputs a place NAME (string), not coordinates. Nominatim resolves the name to authoritative OSM coordinates. Validate that the Nominatim result is within the expected country/region.
**Warning signs:** Events appearing outside the Middle East bounding box.

### Pitfall 4: Vercel Function Timeout During LLM Processing

**What goes wrong:** LLM call takes > 10s (default Hobby timeout), function times out with 504.
**Why it happens:** Default maxDuration is 10s on Hobby. LLM calls to Cerebras are fast (~1-3s for gpt-oss-120b at 1000+ tok/s), but Groq fallback may be slower, and Nominatim geocoding adds latency.
**How to avoid:** Set `maxDuration` to 60s for the events route (Hobby plan maximum). Process LLM + geocoding as a pipeline: LLM call (~3s) + sequential Nominatim calls (~1s each, ~10 events = ~10s) + cache write (~0.5s) = ~15s total. Well within 60s.
**Warning signs:** 504 errors on `/api/events` requests.

### Pitfall 5: Stale LLM Cache Blocking Fresh GDELT Data

**What goes wrong:** LLM-enriched cache is fresh (within 15-min TTL) but the underlying GDELT data has updated. Users see stale events.
**Why it happens:** Two-layer caching (raw GDELT + LLM-enriched) with different staleness windows.
**How to avoid:** The LLM processing always starts from a FRESH GDELT fetch, not from cached GDELT. The LLM-enriched cache and raw GDELT cache have independent TTLs. When serving, prefer LLM cache if fresh; fall back to raw GDELT cache if LLM cache is stale.
**Warning signs:** Events on the map not updating for 30+ minutes.

### Pitfall 6: Breaking the ConflictEventType Discriminated Union

**What goes wrong:** Changing from 11 types to 5 types breaks every file that switches on `entity.type` for conflict events. TypeScript catches compile-time exhaustiveness checks, but runtime checks (string comparisons, CONFLICT_TOGGLE_GROUPS lookups) silently fail.
**Why it happens:** The type union is used in ~15+ files across the codebase.
**How to avoid:** Change the type union FIRST, let TypeScript surface every break. Update CONFLICT_TOGGLE_GROUPS, EVENT_TYPE_LABELS, TYPE_WEIGHTS, GOLDSTEIN_CEILINGS, BASE_CODE_MAP, ROOT_FALLBACK, icons, colors in a single coordinated pass. Run full test suite after each file change.
**Warning signs:** Events not appearing on map, toggles not working, counter rows missing.

## Open Questions

1. **Token Budget with Incremental Processing**
   - What we know: Free tier is 1M TPD (Cerebras) + 200K TPD (Groq). Each batch of ~8 event groups costs ~4K tokens.
   - What's unclear: How many truly NEW event groups appear per 15-minute window? If the overlap with previous windows is high, incremental processing (only process new events) could reduce token usage 5-10x.
   - Recommendation: Build the processing pipeline to diff against cached LLM results and only process new/changed event groups. Store processed event IDs to avoid re-processing.

2. **Nominatim Geocoding Failure Rate for Middle East Place Names**
   - What we know: Nominatim uses OSM data, which has excellent coverage in Iraq, Syria, Iran. The existing reverse geocoder works well. Forward geocoding should work for well-known cities.
   - What's unclear: How well does Nominatim resolve obscure conflict locations (checkpoints, villages, military bases)? What's the failure rate?
   - Recommendation: Implement all three fallback paths from D-14 (Nominatim -> GDELT ActionGeo -> LLM's own estimate) and measure during implementation spike.

3. **gpt-oss-120b Structured Output Reliability**
   - What we know: Cerebras supports strict JSON Schema mode for gpt-oss-120b. Groq has a reported community issue with structured outputs being "ignored" for this model.
   - What's unclear: Is the Groq issue resolved? How reliable is strict mode in practice?
   - Recommendation: Always validate LLM output with Zod post-parse. If Groq strict mode is unreliable, use `json_object` mode + explicit "respond only with JSON" instruction as fallback.

4. **Impact of 60s maxDuration Limit**
   - What we know: Hobby plan maxDuration is 60s. LLM calls are fast (1-3s). Nominatim requires 1s between calls.
   - What's unclear: With batching (2-3 LLM calls) + sequential Nominatim (10-20 calls at 1s each), does the total fit in 60s?
   - Recommendation: Process in two phases. Phase 1: LLM calls (~10s total). Phase 2: Nominatim geocoding for any uncached place names (~15-20s). Total ~30s, well within 60s. If needed, cap Nominatim calls per cycle and geocode remaining in next cycle.
