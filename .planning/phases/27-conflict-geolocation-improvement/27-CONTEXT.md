# Phase 27: Conflict Geolocation Improvement (GDELT Redo) - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

Rearchitect the GDELT conflict event pipeline with LLM-based extraction: precise geolocation via Nominatim, simplified 5-type event ontology, multi-source deduplication/merge, casualty extraction, and situation summaries. GDELT remains the data source; the LLM replaces CAMEO-based classification and ActionGeo-based geolocation with structured extraction from grouped multi-row event bundles.

**Explicitly OUT of scope for Phase 27 (deferred to Phase 27.1):**

- Event search redesign (9-tag vocabulary)
- Event timeline/history tracking
- Source credibility scoring

</domain>

<decisions>
## Implementation Decisions

### NLP Strategy & Architecture

- **D-01:** LLM provider is **Cerebras primary + Groq fallback** (multi-provider adapter, same pattern as flight sources). Cerebras free tier: 1M tokens/day. Groq free tier: 500K tokens/day. Combined 1.5M tokens/day at $0.
- **D-02:** Model is **Llama 3.3 70B** on both providers. Highest quality for structured extraction.
- **D-03:** LLM processing runs as a **Vercel cron background job** every 15 minutes (extends `/api/cron/warm` pattern). The LLM never runs in the `/api/events` request path.
- **D-04:** LLM input is **multiple GDELT rows grouped per real-world event**. The pipeline groups rows that describe the same event (similar timestamp + location + CAMEO) across different sources, then feeds each group as a bundle to the LLM. The LLM synthesizes one canonical record per event.
- **D-05:** LLM output is **flat JSON per event**: `{ location: { lat, lng, name, precision }, type, actors, severity, summary, casualties, sources }`. Simple shape that maps to a modified `MapEntity`.
- **D-06:** When the LLM is unavailable (rate limit, timeout, error), **fall back to raw GDELT data** processed through the current pipeline (CAMEO classification + ActionGeo). Events appear with lower precision but the map never goes blank. Graceful degradation contract.

### Event Type Ontology

- **D-07:** Replace 11 CAMEO-based `ConflictEventType` values with **5 purpose-built categories classified by attack vector**:
  - `airstrike` — airstrikes, drone strikes, aerial bombardment
  - `on_ground` — infantry combat, battles, clashes, firefights
  - `explosion` — shelling, bombing, car bombs, IEDs, rocket attacks
  - `targeted` — assassinations, abductions, targeted raids
  - `other` — blockades, ceasefire violations, mass violence, WMD, posturing
- **D-08:** Toggle design is **master "Events" toggle + 5 indented sub-toggles** (one per type). Replaces current 3-toggle system (Airstrikes, Ground Combat, Targeted).
- **D-09:** Colors remain in the **red spectrum** (distinct shades within the red family). Must not clash with flight (yellow), ship (purple), or site (green).
- **D-10:** The existing `classifyByBaseCode()` CAMEO mapper becomes the **fallback classification** when the LLM is unavailable — maps old CAMEO codes into the new 5-type taxonomy.

### Geolocation Precision

- **D-11:** Target precision is **best-effort with a precision indicator**. Each event gets a `precision` field: `exact` | `neighborhood` | `city` | `region`. The LLM extracts the most specific location name it can from the grouped articles.
- **D-12:** Geocoding uses **Nominatim** (existing adapter at `server/adapters/nominatim.ts`, Redis-cached with 30-day TTL, coordinate quantization). LLM extracts a place name → Nominatim resolves to lat/lng.
- **D-13:** Imprecise events show **radius rings by precision level** on the map: exact = point icon (current), neighborhood = 1km translucent ring, city = 5km ring, region = 25km ring. Visual honesty about uncertainty.
- **D-14:** Nominatim geocoding failure fallback: **defer to spike testing**. The implementation should support three fallback paths (GDELT ActionGeo, drop event, use LLM estimate) so the best option can be chosen empirically during development.
- **D-15:** **Human-readable location name always extracted** by the LLM (e.g., "Deir ez-Zor, eastern Syria" or "near Isfahan nuclear facility"). Shown in tooltips and detail panel alongside coordinates.
- **D-16:** Current centroid ring dispersion system likely obsolete for LLM-processed events. Keep as fallback for raw-GDELT-mode only.

### Palantir-Grade Enrichments

- **D-17:** **Event deduplication/merge is built into the pipeline** — multi-row grouping + LLM synthesis produces one canonical record per real-world event. Source count and list preserved.
- **D-18:** **Situation summary generated by the LLM** for every event — 2-3 sentence human-readable description (e.g., "Israeli airstrike targeted a weapons depot in eastern Damascus at 14:30 UTC, reported by 8 sources including Reuters and Al Jazeera"). Stored in the `summary` field of the flat JSON output.
- **D-19:** **Casualty extraction** — LLM extracts casualty estimates when mentioned in articles. Schema: `casualties: { killed?: number, injured?: number, unknown?: boolean }`. Low marginal cost since the LLM is already processing the text.

### Phase Split

- **D-20:** Phase 27 delivers: LLM pipeline + new event ontology + precision geolocation + dedup/merge + summaries + casualty extraction. **Phase 27.1** (separate phase) delivers: event search redesign (9-tag vocabulary: type, location, actor, precision, severity, near, country, sources, date), event timeline/history, source credibility scoring.

### Claude's Discretion

- LLM prompt engineering (exact system prompt, few-shot examples, JSON schema enforcement)
- Event grouping algorithm (timestamp window, geographic proximity threshold, CAMEO similarity)
- Adapter implementation details (Cerebras SDK vs REST, error retry policy, rate limit backoff)
- Radius ring visual style (color, opacity, border)
- How `classifyByBaseCode()` maps to the new 5-type taxonomy for fallback mode
- GDELT column selection for LLM input (which of the 60+ GDELT CSV columns to include)

</decisions>

<specifics>
## Specific Ideas

- Multi-provider LLM adapter mirrors the flight source adapter pattern (OpenSky → adsb.lol fallback)
- Vercel cron pattern already proven with `/api/cron/warm` — extend it for LLM processing
- Nominatim adapter already exists and is Redis-cached — reuse directly
- Graceful degradation contract: LLM down → serve raw GDELT → same as current behavior. The map never goes blank.
- The 5 event types use the same icon patterns as current (starburst, explosion, crosshair, xmark) plus a new blast wave icon for Explosion
- "Palantir-grade" means: every event has a human-readable summary, precise geocoded location with confidence ring, casualty data when available, and source count. Not just dots on a map.

</specifics>

<deferred>
## Deferred Ideas

### Phase 27.1 (planned)

- **Event search redesign** — 9-tag vocabulary (type, location, actor, precision, severity, near, country, sources, date) replacing the current CAMEO-based tags
- **Event timeline/history** — track event state changes over time (first reported → corroborated → details emerged → casualties updated)
- **Source credibility scoring** — weight sources by reliability (Reuters > random blog), feed into event confidence, render more prominently

### Backlog

- Cross-referencing improved events with site proximity (attack status already exists — benefits automatically from better geolocation)
- Threat density cluster improvements with precise geolocation (clusters become more meaningful)
- Event correlation detection (multiple events forming a pattern/campaign)

</deferred>
