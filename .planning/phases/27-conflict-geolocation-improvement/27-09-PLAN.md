```typescript
/** Map GDELT ActionGeo_Type to precision level for radius rings + tooltip dots */
function actionGeoTypeToPrecision(
  geoType: number | undefined,
): 'exact' | 'neighborhood' | 'city' | 'region' | undefined {
  switch (geoType) {
    case 4:
      return 'exact'; // landmark — most precise GDELT geocoding
    case 3:
      return 'city'; // city-level — ~5km uncertainty
    case 2:
      return 'region'; // ADM1/state — ~25km uncertainty
    case 1:
      return 'region'; // country-level — ~25km uncertainty
    default:
      return undefined; // unknown/missing — no ring shown
  }
}
```

Then in the `normalizeGdeltEvent` function, after the existing `const actionGeoType = ...` line, add:

```typescript
const precision = actionGeoTypeToPrecision(actionGeoType);
```

And add `precision` to the `data` object:

```typescript
data: {
  // ... existing fields ...
  actionGeoType,
  precision,  // <-- ADD THIS LINE after actionGeoType
},
```

This ensures ALL raw GDELT events (not just LLM-enriched ones) carry the `precision` field. The PrecisionRingLayer and EntityTooltip already check for `d.data.precision` — they just find nothing on raw events currently. Once this field is populated, both features activate automatically.

Note: The LLM path in `enrichedToEntities` (events.ts line ~151) overrides this with its own more fine-grained precision (it can detect 'neighborhood' from location names). That's correct: LLM precision is more accurate, so it should override the coarse GDELT mapping.
