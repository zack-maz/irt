# Phase 27.3 Plan 04: Water Label Hygiene — Gap Closure Summary

**Remove the "Dam near unknown" UI failure mode end-to-end: reject low-information facilities at the server filter, sanitize stale cached labels at the client, and make `getWaterFacilityDisplayName` the one place that knows how to title a water facility.**

- **Duration:** ~50 min (two phases: initial worktree executor hit usage limit, inline resumption completed both tasks)

- **Completed:** 2026-04-17T18:37Z
- **Tasks:** 2
- **Files modified:** 12 (11 changed + 1 new)

## Accomplishments

- New `no_city` rejection bucket wired through server interface, Zod schema, test fixture, client interface, and DevApiStatus UI
- Server-side filter rule rejects facilities with no nearestCity (within 150km) AND no wikidata/wikipedia tag
- `labelUnnamedFacilities` no longer writes the literal string 'Unknown' into any label
- New `src/lib/waterLabel.ts` exports `getWaterFacilityDisplayName` as the sole UI entry point for water-facility titles
- All 5 consumer surfaces migrated: WaterTooltip, counter rows, proximity alerts, breadcrumb, detail-panel image alt
- Pre-fix "near Unknown" sentinel regex (`\bnear\s+unknown\s*$/i`) sanitizes stale cached labels at read time

## Decisions Made

- The rejection rule runs AFTER `findNearestCity`/`linkRiver` because it needs their output; re-ordering in `normalizeWaterElement` is intentional.
- Wikidata/wikipedia tagged facilities keep their slot even when nearestCity is null — remote-but-notable infrastructure is not noise.
- The server label guard AND client helper sentinel regex are deliberately redundant — defense in depth against stale Redis entries from before this fix.
- `WATER_TYPE_LABELS` unified on `'Desalination Plant'` (matches server) — closes IN-04 drift between server and client vocabularies.
