## Gap Closure

### G-01 / WR-02 — Blocking: water route test regression

- **Root cause (from VERIFICATION.md):** Plan 01 added `waterResponseSchema.extend({ filterStats: waterFilterStatsSchema })` but the Zod schema requires 5 sub-fields when `filterStats` is present. The test mock returned `stats: {}` — an empty object that passes TypeScript but fails the Zod validator. Under `NODE_ENV=test`, `validateResponse` throws `RESPONSE_SCHEMA_MISMATCH` (500), so Test 1's `expect(body.stale).toBe(false)` received `true` from the error-fallback path.
- **Fix:** Added `emptyStats` fixture at module scope in `server/__tests__/routes/water.test.ts` with all 5 required sub-fields zero/empty. Replaced three `stats: {}` call sites with `emptyStats`. Narrowed mock return type from `stats: unknown` to `stats: typeof emptyStats` (IN-03 addressed as a drive-by).
- **Commit:** `8317960`
- **Outcome:** `water.test.ts` 5/6 -> 6/6 passing. Full suite baseline: 1398 pass / 53 fail -> 1399 pass / 52 fail. No new failures; the 52 remaining failures are pre-existing and documented in Plan 02 SUMMARY.

### WR-01 — Consistency: REV-5 attack-event set inconsistency across three consumers

- **Root cause (from REVIEW.md):** Phase 27.3 Plan 02 expanded the attack-event set in `useWaterLayers.ts` (`ATTACK_EVENT_TYPES = {airstrike, explosion, targeted, on_ground}`) but left two sibling consumers on the pre-REV-5 narrow literal `new Set(['airstrike', 'explosion'])`. Map showed black for facilities near `targeted`/`on_ground` events, but `WaterFacilityDetail.tsx` reported `isDestroyed=false` (score = healthToScore, not 0) and `useCounterData.ts` reported the health-based score in the counter dropdown.
- **Fix:** New file `src/lib/waterAttackEvents.ts` exports `WATER_ATTACK_EVENT_TYPES`. All three consumers import it:
  - `useWaterLayers.ts` — removed inline `DESTRUCTIVE_EVENT_TYPES`, `COMBAT_EVENT_TYPES`, `ATTACK_EVENT_TYPES`; uses `WATER_ATTACK_EVENT_TYPES.has(e.type)` in the destructive event filter.
  - `WaterFacilityDetail.tsx` — removed narrow `DESTRUCTIVE_EVENT_TYPES`; `isDestroyed` check uses the shared set.
  - `useCounterData.ts` — removed narrow `DESTRUCTIVE`; destroyed-water loop uses the shared set.
- **Commit:** `81469c7`
- **Outcome:** `grep` checks confirm exactly 4 files reference `WATER_ATTACK_EVENT_TYPES` (1 definition + 3 consumers), zero remaining narrow literals, zero remaining `DESTRUCTIVE_EVENT_TYPES` / `ATTACK_EVENT_TYPES` local declarations.

## Decisions Made

See `key-decisions` in frontmatter. Highlights:

- **Why a new file rather than extending `src/lib/attackStatus.ts`:** `attackStatus.ts` operates on SiteEntity shapes via `computeAttackStatus`; water facilities have their own shape and attack semantics. Keeping `waterAttackEvents.ts` narrowly scoped makes the ownership boundary clear and avoids coupling unrelated consumers.
- **Why `typeof emptyStats` (not importing `WaterFilterStats`):** The mock file already avoids importing from the adapter internals; `typeof emptyStats` keeps that boundary while still guaranteeing the mock return type follows the fixture shape. If the fixture shape drifts from the Zod schema, the route's `sendValidated` will reject it at test time and the test will fail — the type system tells us about the drift at edit time.
- **Plan deviation rules considered but not applied:** The plan's deviation guide noted the option of keeping `WaterFacilityDetail.tsx` on a narrower subset for Attack History display. Rejected: the semantic widening is the deliberate REV-5 fix, and three-way consistency is the stated goal. A facility near an `on_ground` event is correctly flagged as attacked everywhere now.
