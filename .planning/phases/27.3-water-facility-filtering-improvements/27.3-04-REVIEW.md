## Warnings

### WR-01: Search autocomplete still reads raw `w.label` — bypasses `getWaterFacilityDisplayName`

**File:** `src/lib/tagRegistry.ts:113`
**Issue:** The `getNearValues` function in the Cmd-K tag autocomplete registry ingests
water facility labels directly:

```ts
for (const w of data.water) {
  if (w.label) names.push({ value: w.label, count: 1 });
}
```

This path was not in the plan's consumer migration list but it IS a user-facing
display surface: the `near:` tag autocomplete dropdown. Any stale Redis cache entry
still carrying a pre-fix `"Dam near Unknown"` label will surface as a suggestion in
the search popup, contradicting the plan's first must-have truth ("No water facility
ever renders a user-facing string containing the literal token 'unknown'... from any
client-side fallback"). The `NEAR_UNKNOWN_RE` sentinel sanitization intentionally
lives only inside `getWaterFacilityDisplayName`, so a direct `w.label` read does not
benefit from it.

**Fix:** Migrate the tagRegistry path to the helper — it is the same two-line change
as the other consumers:

```ts
import { getWaterFacilityDisplayName } from '@/lib/waterLabel';
// ...
if (data.water) {
  for (const w of data.water) {
    const name = getWaterFacilityDisplayName(w);
    if (name) names.push({ value: name, count: 1 });
  }
}
```

Keeping the `if (w.label)` guard is no longer necessary because
`getWaterFacilityDisplayName` always returns a non-empty string.

## Info

### IN-01: `WATER_TYPE_LABELS` drift in `src/types/ui.ts` still reads "Desalination"

**File:** `src/types/ui.ts:122-126`
**Issue:** The SUMMARY claims IN-04 (the drift between `'Desalination'` and
`'Desalination Plant'`) is "closed as side-effect of this plan," and
`waterLabel.ts`, `WaterOverlay.tsx`, and `WaterFacilityDetail.tsx` all adopted
`'Desalination Plant'`. However `src/types/ui.ts` still exports:

```ts
export const WATER_TYPE_LABELS: Record<string, string> = {
  dam: 'Dam',
  reservoir: 'Reservoir',
  desalination: 'Desalination', // <-- still the old wording
};
```

That export is consumed by `src/components/layout/FilterPanelSlot.tsx:20,551`, so the
Filter Panel's dropdown still shows "Desalination" while the tooltip, counter, detail
panel, and breadcrumb now show "Desalination Plant". IN-04 is partially closed, not
fully.

**Fix:** Update `src/types/ui.ts:125` to `desalination: 'Desalination Plant'` so the
filter-panel vocabulary matches the other four surfaces, OR leave it intentionally
shorter (filter panels prefer concise labels) and document the intentional divergence
in `waterLabel.ts` so a future developer does not "fix" it.

### IN-02: `'Unknown'` literal remains in `WaterFacilityDetail.tsx` Operator field

**File:** `src/components/detail/WaterFacilityDetail.tsx:113`
**Issue:**

```tsx
<DetailValue label="Operator" value={facility.operator || 'Unknown'} />
```

The plan's must-have truth is scoped to "display name," and the Operator field is
distinct from the display name — so this is out of the strict invariant. But it is
the only remaining `'Unknown'` literal on any water UI surface, and the plan's
overall posture is "show fewer high-quality items rather than pad with placeholders."
A missing operator is arguably better rendered as an em-dash or hidden row than as
the word "Unknown." Flagging as info because (a) it is consistent with the Site
detail panel's convention and (b) it was not in the plan's scope.

**Fix:** Optional — either leave as-is (consistent with Site panel) or mirror the
"hide row when absent" pattern used elsewhere:

```tsx
{
  facility.operator && <DetailValue label="Operator" value={facility.operator} />;
}
```

### IN-03: Unused `typeLabel` allocation in early-return branch of `getWaterFacilityDisplayName`

**File:** `src/lib/waterLabel.ts:41-46`
**Issue:** `const typeLabel = WATER_TYPE_LABELS[facility.facilityType];` is computed
at the top of the function but is never used when the first branch (raw label
non-sentinel) returns. This is a micro-micro inefficiency (a single object property
read per facility per render) and is arguably clearer this way — flagging only for
completeness.

**Fix:** None required. If pedantry demands it, move the `typeLabel` calculation
below the first-branch early return:

```ts
export function getWaterFacilityDisplayName(facility: WaterFacility): string {
  const raw = facility.label?.trim();
  if (raw && !NEAR_UNKNOWN_RE.test(raw)) {
    return raw;
  }
  const typeLabel = WATER_TYPE_LABELS[facility.facilityType];
  // ... rest unchanged
}
```
