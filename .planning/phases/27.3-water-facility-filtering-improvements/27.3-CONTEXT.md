# Phase 27.3: Water Facility Filtering Improvements

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Improve the quality and accuracy of water facility data pulled from Overpass. Fix broken queries (dams using wrong OSM tag), reduce noise (5k reservoirs overwhelming the map), add dev diagnostics for data-driven tuning, and enrich facilities with capacity/population/upstream data. Drop treatment plants (near-zero ME coverage). Make the water layer Palantir-grade using only free/public APIs.

</domain>

<decisions>
## Implementation Decisions

### 1. Fix Dam Query Tags (Critical Bug)

- **D-01:** Current `waterway=dam` query returns only 1 result because it targets the waterway node tag, not the structure. Switch to `man_made=dam` (and potentially `waterway=dam` as union) to catch actual dam structures. This is the single highest-impact fix.

### 2. Reservoir Noise Reduction

- **D-02:** Claude's discretion on filtering approach. The 5k reservoir count must come down dramatically. Recommended hybrid: apply notability filter (wikidata/wikipedia or named) to ALL countries including priority ones, plus investigate area-based thresholds from OSM geometry if available. Target: 50-200 strategically relevant reservoirs, not 5k.

### 3. Drop Treatment Plants

- **D-03:** Remove `treatment_plant` as a facility type entirely. OSM coverage in ME is near-zero. Remove the query, the type from `WaterFacilityType`, the filter toggle, and the icon. Clean removal, not a stub.

### 4. Dev Diagnostics for Filter Tuning

- **D-04:** Add water facility filter statistics to the DevApiStatus panel. Show: raw Overpass count per type, post-filter count, rejection reasons/counts. Enables data-driven threshold tuning without checking server logs.

### 5. Query Structure Improvements

- **D-05:** Claude's discretion on query robustness. May include: additional tag combinations for under-represented types, smarter dedup (same physical facility as node + way + relation), regional query splitting for error isolation, expanded tag coverage. Goal: catch real facilities that current queries miss.

### 6. Facility Enrichment — Capacity/Volume

- **D-06:** Pull OSM capacity-related tags where available (dam height, reservoir volume/area, plant throughput). Most ME facilities won't have these, but the ones that do become significantly richer. Store as optional fields on WaterFacility.

### 7. Facility Enrichment — Population Served

- **D-07:** Cross-reference facility location with nearby city populations from me-cities.json. Approximate impact radius — e.g. "serves ~2M people in greater Damascus area". Enriches the detail panel and severity scoring context.

### 8. Facility Enrichment — Upstream/Downstream River Links

- **D-08:** Link facilities to rivers they sit on (via OSM waterway relations or spatial proximity to our existing rivers.json data). Shows cascading risk — "this dam feeds the Euphrates which supplies 3 downstream facilities". May be partial coverage but even partial is valuable.

### Claude's Discretion

- Exact notability threshold for reservoirs (area-based, name-based, or hybrid)
- Additional OSM tag combinations to try for dams and desalination
- Dedup strategy for multi-representation facilities
- How to compute/display population served estimates
- River linkage algorithm (relation-based vs proximity-based)

</decisions>
