### 2. DevApiStatus Water Filters diagnostics panel renders

expected: |
Open the dev server, hit /api/water fresh (not from Redis/dev cache — clear Redis or
delete `.dev-cache/water.json` first). Open the DevApiStatus overlay. The "Water Filters"
section renders showing: raw vs filtered counts per facility type (dam/reservoir/desalination),
rejection tally (excluded_location, not_notable, no_name, duplicate, low_score),
enrichment coverage (cap/city/river), and a 5-bucket score histogram. Panel is absent
on cached responses (which is expected — filterStats only populates on fresh fetch).
result: pass
note: |
Initially reported as 'Water is stuck at INIT' — user confirmed on follow-up that water
did load, Overpass fetch just took longer than expected. INIT state lingers during the
initial fetch (can be 30-90s for a cold Overpass query) which reads as 'stuck' until
the user waits it out. Functionality is correct; UX follow-up candidate: surface
loading progress more clearly (spinner / elapsed-time hint) while the initial query runs.

### 3. WaterFacilityDetail enrichment sections render

expected: |
Click a large water facility near a major river — e.g. Mosul Dam (Tigris),
Tabqa Dam (Euphrates), or Aswan High Dam (Nile). Detail panel shows the existing
Water Stress / Aqueduct sections PLUS new sections (when OSM data is present):
Capacity (height in m / volume in million m³ / area in km²), Population Served
(nearest city + distance), and River System (linked river + "Damage to this facility
may impact downstream X users" note). In dev mode, a Notability Score section
(0-100) also appears.
result: issue
reported: "Some of these dams are unnecessary... it just says 'Dam near unknown'. The detail panel is working great"
severity: major
note: |
Detail panel itself renders correctly (pass on the panel criterion). But the user
surfaced a data-quality issue with the filter's output: low-quality dams are slipping
through with fallback label 'Dam near unknown'. This means either (a) the facility
has no OSM name AND no resolvable nearest city, yet still passed the holistic filter;
or (b) the label-generation path is producing 'unknown' when nearestCity enrichment
fails. Either way, the phase goal 'reduce reservoir noise to 50-200 via HOLISTIC filter'
is not fully achieved — named+priority-country dams with no city context shouldn't
be shown, or at minimum shouldn't be labeled 'near unknown'.

on_ground event should render as SOLID BLACK (attacked state) on the map, overriding
the normal stress-gradient color. Before the REV-5 fix, facilities were only blackened
