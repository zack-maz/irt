# Phase 27.3: Water Facility Filtering Improvements - Research

**Researched:** 2026-04-15
**Domain:** OpenStreetMap Overpass queries, water infrastructure data quality, facility enrichment
**Confidence:** HIGH

## Summary

Phase 27.3 focuses on improving the quality of water facility data from Overpass API. The primary issues are: (1) the dam query uses `waterway=dam` which is actually the **correct** OSM tag (not wrong as initially suspected) -- the low dam count is caused by aggressive post-query filtering, not the tag itself; (2) reservoirs return ~5,000 results and need better filtering to reach a 50-200 target; (3) treatment plants have near-zero ME coverage and should be removed entirely; (4) facilities need enrichment with capacity/population/river-link data; and (5) a dev diagnostics panel should expose filter statistics.

Key finding: Overpass returns 8,692 `waterway=dam` elements in the ME bbox (verified via live query). The "only 1 result" problem reported by the user is a **filtering issue**, not a tagging issue. The `isExcludedLocation()` and `hasName()` filters are overly aggressive for dams. The fix is to relax dam filtering thresholds, not change the tag. Additionally, `man_made=dam` is NOT an established OSM tag -- the wiki documents only `waterway=dam` as the standard.

**Primary recommendation:** Fix dam filtering (relax notability requirements since 8.6k raw dams exist), implement area-based reservoir filtering (use Overpass `out geom` to calculate polygon area), cleanly remove treatment_plant type, add enrichment fields to WaterFacility, and build diagnostics into DevApiStatus.

### Pattern 2: Reservoir Area-Based Filtering

**What:** Use Overpass `out geom` to get polygon geometry, compute approximate area from bounding box, filter by area threshold.
**When to use:** For reservoir queries where volume is too high and name/notability filtering alone is insufficient.
**Example:**

```typescript
// Source: [VERIFIED: Overpass API documentation]
// Approximate area from bounding box (not exact polygon area, but sufficient for filtering)
function approximateAreaKm2(coords: number[][]): number {
  if (coords.length < 3) return 0;
  let minLat = Infinity,
    maxLat = -Infinity;
  let minLng = Infinity,
    maxLng = -Infinity;
  for (const [lng, lat] of coords) {
    minLat = Math.min(minLat, lat);
    maxLat = Math.max(maxLat, lat);
    minLng = Math.min(minLng, lng);
    maxLng = Math.max(maxLng, lng);
  }
  const latDist = (maxLat - minLat) * 111.32; // km per degree latitude
  const avgLat = (minLat + maxLat) / 2;
  const lngDist = (maxLng - minLng) * 111.32 * Math.cos((avgLat * Math.PI) / 180);
  return latDist * lngDist * 0.7; // 0.7 factor approximates polygon vs bbox ratio
}
```

**Tradeoff:** `out geom` returns much larger payloads than `out center tags`. For 5k reservoirs, geometry data could be 10-50MB. Alternative: use `out center tags` plus the `bounds` output mode which gives just the bounding box.

### Pattern 3: Proximity-Based River Linkage

**What:** Link facilities to rivers by spatial proximity to rivers.json line features.
**When to use:** When OSM waterway relations are incomplete (which they are in ME).
**Example:**

```typescript
// Source: Existing rivers.json has 6 rivers with MultiLineString geometry
function findNearestRiver(
  facilityLat: number,
  facilityLng: number,
  rivers: RiverFeature[],
): { name: string; distanceKm: number } | null {
  let nearest: { name: string; distanceKm: number } | null = null;
  for (const river of rivers) {
    const coords = flattenCoords(river.geometry);
    for (const [lng, lat] of coords) {
      const d = haversine(facilityLat, facilityLng, lat, lng);
      if (!nearest || d < nearest.distanceKm) {
        nearest = { name: river.properties.name, distanceKm: d };
      }
    }
  }
  // Only link if within reasonable distance (e.g., 20km)
  return nearest && nearest.distanceKm <= 20 ? nearest : null;
}
```

### Anti-Patterns to Avoid

- **Changing `waterway=dam` to `man_made=dam` only:** The OSM wiki documents `waterway=dam` as the standard tag. `man_made=dam` is not formally standardized. Always include `waterway=dam` in the query. [VERIFIED: OSM wiki]
- **Using `out geom` for ALL facility types:** Only reservoirs need area-based filtering. Dams and desalination can stay with `out center tags` (much smaller payload).
- **Computing exact polygon area server-side:** Bounding box approximation is sufficient for filtering. Exact area computation (Shoelace formula on geodesic coordinates) is overkill for a threshold check.
- **Fetching all 5k reservoir geometries in one query:** The reservoir query already times out at 30s. Consider limiting to `["name"]` OR `["wikidata"]` reservoirs for the geometry query, then applying area filter.

## Common Pitfalls

### Pitfall 1: Overpass Rate Limiting on Geometry Queries

**What goes wrong:** Requesting `out geom` for thousands of reservoirs creates a massive response that triggers Overpass rate limiting or timeouts.
**Why it happens:** `out geom` includes full coordinate arrays for every polygon node. 5k reservoirs could be 50MB+ of JSON.
**How to avoid:** Only request geometry for the reservoir query (not dams/desalination). Pre-filter with `["name"]` or `["wikidata"]` in the Overpass query to reduce count before requesting geometry. Or use `out bb` (bounding box only) which is much lighter.
**Warning signs:** Reservoir query timeout (currently 90s), HTTP 429 from Overpass.

### Pitfall 2: Treatment Plant Removal Leaves Orphaned References

**What goes wrong:** Removing `treatment_plant` from the type union breaks code that still references it.
**Why it happens:** `treatment_plant` is referenced in ~40 files across server and client (type definitions, filter toggles, icon mappings, counter components, test files, OpenAPI spec).
**How to avoid:** Grep for all references before removal. Use TypeScript compiler errors as a safety net -- removing from the union type will flag all broken references.
**Warning signs:** TypeScript build errors, runtime crashes on water layer rendering.

### Pitfall 3: Existing Test Failures

**What goes wrong:** Two tests in `overpass-water.test.ts` are already failing. The test at line 285 expects `treatment_plant` to be null in non-priority countries, but the current code's `hasName` filter allows it through with a name tag.
**Why it happens:** Tests weren't updated when 27.2 changed the filtering logic.
**How to avoid:** Fix the two failing tests FIRST before making any changes. Since treatment_plant is being removed entirely, these tests should be deleted rather than fixed.
**Warning signs:** `npx vitest run server/__tests__/adapters/overpass-water.test.ts` shows 2 failures.

### Pitfall 4: Dam Count Misattribution

**What goes wrong:** The "only 1 dam" problem is attributed to the wrong tag (`waterway=dam` vs `man_made=dam`), leading to a tag change that misses the real fix.
**Why it happens:** `waterway=dam` returns 8,692 elements in the ME bbox (verified). The low count after filtering is because `isExcludedLocation()` and `hasName()` are too aggressive for dams in non-priority countries, and many ME dams lack wikidata/wikipedia tags.
**How to avoid:** The fix is to relax the notability filter for dams (use `hasName` for ALL countries, not just priority ones) and ensure `isExcludedLocation` doesn't falsely exclude Iran/Pakistan (which was already fixed in 27.2 per STATE.md).
**Warning signs:** After the "fix", dam count doesn't improve significantly.

### Pitfall 5: me-cities.json Does Not Exist

**What goes wrong:** CONTEXT.md references `me-cities.json` for population enrichment, but this file does not exist in the codebase.
**Why it happens:** `me-cities.json` was a planned artifact from Phase 27 (Conflict Geolocation Improvement) which hasn't been executed yet. The STATE.md mentions "me-cities.json as single source of truth" but this was from the scrapped Phase 26.2.
**How to avoid:** Use the existing `CITY_CENTROIDS` array (42 entries) from `server/lib/geoValidation.ts` for population proximity. If richer data is needed, create a minimal `me-cities.json` as part of this phase, or use GeoNames API (already referenced in STATE.md with 200k population threshold).
**Warning signs:** Import errors for non-existent file.

### Key Observations from Live Data

1. **Dams are NOT underpulling at the query level.** 8,692 raw elements exist. The problem is post-query filtering.
2. **Named dams = 1,766.** The `hasName` filter is reasonable for non-priority countries but needs relaxation for priority countries.

4. **Reservoir query is expensive.** Even `out count` times out at 30s. The current 90s timeout is necessary and geometry queries will need careful design.

## Open Questions

1. **Reservoir area threshold**
   - What we know: 5k reservoirs is too many, target is 50-200. Named reservoirs are ~1,766 (still too many). Notable (wikidata) reservoirs likely ~50-100.
   - What's unclear: Exact area threshold in km2 to use. Does Overpass support `out bb` for bounding-box-only output?
   - Recommendation: Start with strict notability filter (wikidata/wikipedia) for ALL countries (including priority), then relax if count is too low. Area filtering is a backup, not primary.

2. **`man_made=dam` tag prevalence**
   - What we know: OSM wiki documents only `waterway=dam`. Some informal use of `man_made=dam` exists.
   - What's unclear: How many elements in ME bbox use `man_made=dam` without also having `waterway=dam`.
   - Recommendation: Include in union query (costs nothing), but don't expect significant gains.

3. **Population data source for D-07**
   - What we know: `me-cities.json` doesn't exist. CITY_CENTROIDS has 42 entries without population numbers. GeoNames 200k threshold was previously discussed.
   - What's unclear: Whether CITY_CENTROIDS alone is sufficient or if we need a richer dataset.
   - Recommendation: Add approximate population to CITY_CENTROIDS entries (manual, ~42 values) or create a minimal JSON file with name/lat/lng/population for the 42 cities.
