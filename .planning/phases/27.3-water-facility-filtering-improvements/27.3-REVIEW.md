# Phase 27.3: Code Review Report

**Reviewed:** 2026-04-17T12:00:00Z
**Depth:** standard
**Files Reviewed:** 26
**Status:** issues_found

## Summary

Phase 27.3 ships a well-structured server-side holistic filtering pipeline
with enrichment, dev file cache fallback, and clean client-side
treatment_plant removal. The `WaterFilterStats` plumbing from adapter →
route → client store → DevApiStatus is consistent, and the optional
`filterStats` envelope field is correctly serialized (Zod schema, OpenAPI,
TypeScript type).

The review found **no critical issues**. Six warnings cluster around one
coherent regression: the phase claims to expand `DESTRUCTIVE_EVENT_TYPES`
to include `'targeted'` and `'on_ground'` for attacked-facility detection
(REV-5), but the expansion was only applied in `useWaterLayers.ts`. Two
parallel consumers (`WaterFacilityDetail.tsx` and `useCounterData.ts`)
still use the old narrow `new Set(['airstrike', 'explosion'])`, which
means the map overlay, the counter dropdown, and the detail panel all
disagree about which facilities are "attacked." Remaining warnings cover
a test payload that violates the new Zod schema, a stale comment, and a
dev-mode console.debug that runs on every memo recompute.

Info items are minor — naming drift, unused canvas slots, and a couple
of cross-cutting concerns that fall outside the phase's scope but are
worth flagging.

## Critical Issues

_None._

## Warnings

### WR-01: REV-5 attacked-facility detection is inconsistent across three consumers

**File:** `src/hooks/useWaterLayers.ts:26`, `src/components/detail/WaterFacilityDetail.tsx:39`, `src/components/counters/useCounterData.ts:285`
**Issue:** The phase context (and `27.3-02-SUMMARY.md`) states that REV-5
expanded `DESTRUCTIVE_EVENT_TYPES` to include `'targeted'` and
`'on_ground'`. This expansion was applied only in `useWaterLayers.ts`,
where `ATTACK_EVENT_TYPES = new Set(['airstrike', 'explosion',
'targeted', 'on_ground'])` correctly classifies destroyed facilities for
the map overlay.

However, the two other consumers still use the pre-REV-5 narrow set:

- `WaterFacilityDetail.tsx:39` — `const DESTRUCTIVE_EVENT_TYPES = new Set(['airstrike', 'explosion']);`
- `useCounterData.ts:285` — `const DESTRUCTIVE = new Set(['airstrike', 'explosion']);`

Practical consequence: a facility near a `'targeted'` or `'on_ground'`
event will render as black (destroyed) on the map with score 0/10, but
the detail panel's "Attack History" will either omit it (`isDestroyed`
false) or show different data, and the Counters dropdown score will use
`healthToScore(f.stress.compositeHealth)` instead of 0. This fractures
the three views of the same facility's health state.

**Fix:** Promote the attack-type set to a single shared constant (for
example, in `src/lib/attackStatus.ts` alongside `computeAttackStatus`) and
import it from all three consumers:

```ts
// src/lib/waterAttack.ts (or extend src/lib/attackStatus.ts)
export const WATER_ATTACK_EVENT_TYPES = new Set([
  'airstrike',
  'explosion',
  'targeted',
  'on_ground',
] as const);
```

Then in the three consumers:

```ts
// useWaterLayers.ts
import { WATER_ATTACK_EVENT_TYPES } from '@/lib/waterAttack';
const destructiveEvents = events.filter(
  (e) => WATER_ATTACK_EVENT_TYPES.has(e.type) && e.timestamp <= dateEnd,
);

// WaterFacilityDetail.tsx
import { WATER_ATTACK_EVENT_TYPES } from '@/lib/waterAttack';
const isDestroyed = attacks.some((e) => WATER_ATTACK_EVENT_TYPES.has(e.type));

// useCounterData.ts
import { WATER_ATTACK_EVENT_TYPES } from '@/lib/waterAttack';
const destructive = allEvents.filter(
  (e) => WATER_ATTACK_EVENT_TYPES.has(e.type) && e.timestamp <= dateEnd,
);
```

### WR-02: Water route test mocks return a stats payload that fails Zod validation

**File:** `server/__tests__/routes/water.test.ts:14-19, 161, 185`
**Issue:** The test mocks `fetchWaterFacilities` to return
`{ facilities: [...], stats: {} }`. The route handler in
`server/routes/water.ts:142-152` destructures this and passes the empty
`{}` object through `sendValidated(res, waterResponseSchema, {...,
filterStats })`. `waterFilterStatsSchema` is declared `.optional()`, but
when present it requires `rawCounts`, `filteredCounts`, `rejections`,
`enrichment`, and `scoreHistogram` (none of which are optional). An empty
object is therefore a schema violation.

In `validateResponse.ts:42`, when `NODE_ENV !== 'production'` a schema
mismatch throws an `AppError(500, 'RESPONSE_SCHEMA_MISMATCH', ...)`.
Vitest sets `NODE_ENV=test` by default, which is not `'production'` —
so the test path that exercises a successful fetch (lines 184-200)
should return a 500 rather than the asserted `res.ok === true`.

If these tests are currently passing, it is only because `config.ts` mock
suppresses `NODE_ENV` or the schema check is skipped for some other
reason — worth verifying. Either way, the mock should return a
well-formed stats object or explicitly `undefined` to match production
behavior.

**Fix:** Either return `undefined` (preferred, and only supply a real
`stats` object in the one test that asserts on filterStats), or construct
a valid stub:

```ts
const emptyStats = {
  rawCounts: {},
  filteredCounts: {},
  rejections: {
    excluded_location: 0,
    not_notable: 0,
    no_name: 0,
    duplicate: 0,
    low_score: 0,
  },
  enrichment: { withCapacity: 0, withCity: 0, withRiver: 0 },
  scoreHistogram: [],
};
mockFetchWaterFacilities.mockResolvedValue({
  facilities: [sampleFacility],
  stats: emptyStats,
});
```

### WR-03: `console.debug` in useWaterLayers runs on every memo recompute

**File:** `src/hooks/useWaterLayers.ts:179-188`
**Issue:** Inside a `useMemo` callback, the hook logs an object containing
`Array.from(new Set(events.map((e) => e.type)))` every time the memo
recomputes. The memo's dependency array is wide (`events`, `facilities`,
`dateEnd`, and nine filter toggles), so this fires on each slider tick or
toggle change. The `Array.from(new Set(...))` also allocates a new Set
and Array per log, amplifying GC pressure in dev.

The comment says "Log once per render" but the implementation does not
actually guard against repeated logging within a session.

**Fix:** Either remove the log (it is coarse diagnostic-only), use a
`useRef`-gated "log once on first populated value", or move the signal
into `DevApiStatus` where it is more controllable:

```ts
// option A: remove entirely
// option B: guard with a ref
const loggedRef = useRef(false);
if (import.meta.env.DEV && !loggedRef.current && events.length > 0) {
  loggedRef.current = true;
  console.debug('[useWaterLayers] attack detection initial sample:', {
    totalEvents: events.length,
    destructiveEvents: destructiveEvents.length,
    eventTypes: Array.from(new Set(events.map((e) => e.type))),
    facilities: filteredFacilities.length,
    dateEnd: new Date(dateEnd).toISOString(),
  });
}
```

### WR-05: `extractCapacityTags` accepts malformed volume like "abc123"

**File:** `server/adapters/overpass-water.ts:341-357`
**Issue:** `parseFloat('abc123')` returns `NaN`, which is correctly
rejected. But `parseFloat('123abc')` returns `123`, and
`parseFloat('1.5e308 m3')` returns `1.5e308`. For OSM `height`, `volume`,
or `area`, a malformed value such as `"capacity: 10,000,000"` parses as
`10` (parseFloat stops at the first comma), silently understating volume
by six orders of magnitude. This will later be rendered in
`WaterFacilityDetail.tsx:192-196` as `"0.0 M m³"`.

**Fix:** Add a sanity range and require the value to be a plausible
finite positive number. Optionally sniff common suffix units (km3 →
multiply by 1e9):

```ts
const stripUnits = (v: string): number | undefined => {
  const n = parseFloat(v);
  if (!Number.isFinite(n) || n <= 0) return undefined;
  return n;
};
const height = heightRaw ? stripUnits(heightRaw) : undefined;
// later:
if (height !== undefined) result.height = height;
```

(Drop the `!isNaN` checks since `Number.isFinite(NaN)` is already false.)

### WR-06: `saveDevWaterCache` writes the filterStats blob every successful fetch, doubling dev-cache disk churn

**File:** `server/routes/water.ts:145`
**Issue:** The dev cache is written with
`saveDevWaterCache({ facilities, stats: filterStats })`, so every cold
fetch rewrites the whole JSON. When the cache is later loaded at
`loadDevWaterCache`, only `devCached.facilities` is consumed — the
`stats` field is never re-used (it is deliberately synthesized fresh
only on cold fetch). Persisting stats to disk is therefore dead payload
and the type is intentionally `unknown` on both sides.

This is not a correctness bug, but it is a minor observability smell:
disk snapshots capture per-run stats that will mislead anyone inspecting
the file, and it inflates the cache file by a small but nonzero amount.

**Fix:** Save just the facility payload and narrow the type:

```ts
saveDevWaterCache<WaterFacility[]>(facilities);

// devFileCache.ts: the generic already works; just call sites change.
// load site:
const devCached = loadDevWaterCache<WaterFacility[]>();
if (devCached) {
  const labeled = await labelUnnamedFacilities(devCached);
  ...
}
```
