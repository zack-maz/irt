### Gaps

#### G-01: `server/__tests__/routes/water.test.ts` — New test regression (WR-02)

**Status:** FAILED

**Root cause:** Plan 01 introduced `waterResponseSchema.extend({ filterStats: waterFilterStatsSchema })` and changed `fetchWaterFacilities` return type to `{ facilities, stats }`. The route handler passes `stats` to `sendValidated`, which validates against `waterFilterStatsSchema` requiring all 5 sub-fields.

The test mock at lines 17, 161, and 185 returns `stats: {}` (empty object). When the route calls `sendValidated`, the Zod schema check fails with a `RESPONSE_SCHEMA_MISMATCH` error and returns a 500. The test asserts `body.stale === false` but receives the error-fallback response with `stale: true`.

Plan 02's self-check claimed "all 53 pre-existing failures" but this failure was introduced by Plan 01 and is the 54th (confirmed by the code review WR-02 filed 2026-04-17T12:00:00Z after the summary at 07:29Z).

**Fix:** In `server/__tests__/routes/water.test.ts` lines 17, 161, 185:

```ts
const emptyStats = {
  rawCounts: {},
  filteredCounts: {},
  rejections: { excluded_location: 0, not_notable: 0, no_name: 0, duplicate: 0, low_score: 0 },
  enrichment: { withCapacity: 0, withCity: 0, withRiver: 0 },
  scoreHistogram: [],
};
mockFetchWaterFacilities.mockResolvedValue({ facilities: [sampleFacility], stats: emptyStats });
```

Or return `stats: undefined` (the `filterStats` field is `.optional()` in the schema so `undefined` passes validation).

### WR-01 Warning: REV-5 fix partially rolled out (not a blocker — map is correct)

The code review identified that the REV-5 attack-event expansion was applied in **useWaterLayers.ts** (the map overlay) but NOT in two secondary consumers:

| Consumer                     | Set                                                                | Status                  |
| ---------------------------- | ------------------------------------------------------------------ | ----------------------- |
| `useWaterLayers.ts:26-29`    | `ATTACK_EVENT_TYPES = {airstrike, explosion, targeted, on_ground}` | CORRECT (REV-5 applied) |
| `WaterFacilityDetail.tsx:39` | `DESTRUCTIVE_EVENT_TYPES = new Set(['airstrike', 'explosion'])`    | NARROW (pre-REV-5)      |
| `useCounterData.ts:285`      | `DESTRUCTIVE = new Set(['airstrike', 'explosion'])`                | NARROW (pre-REV-5)      |

**Impact assessment:** The map overlay is the primary surface for the phase goal "fix the invisible-attacked-water-sites bug." A facility near a `targeted` or `on_ground` event will correctly render as solid black on the map. However, the detail panel's `isDestroyed` flag (used to show score=0 and the "destroyed" badge) will evaluate to false for targeted/on_ground events, and the counter dropdown score will use `healthToScore` instead of 0.

This is an inconsistency — three views of the same facility give different health state answers — but it does NOT block the stated phase goal. The bug "invisible attacked water sites" is fixed at the map layer. The labeling inconsistency in detail panel and counter dropdown is a follow-up issue.

**Recommended fix for a follow-up plan:** Extract a shared `WATER_ATTACK_EVENT_TYPES` constant from `src/lib/attackStatus.ts` or `src/lib/waterAttack.ts` and import it in all three consumers (per code review WR-01 suggestion).
