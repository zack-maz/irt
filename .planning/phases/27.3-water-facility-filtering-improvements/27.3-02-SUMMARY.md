## Accomplishments

- Client-side `treatment_plant` is gone — zero matches in `src/`, one historical note in docs. TypeScript compiles clean; icons, counters, labels, filters, detail panel, dev panel all aligned to the 3-type WaterFacilityType.
- REV-5 (attacked water facilities invisible) root-caused and fixed: the destructive-event set was `['airstrike', 'explosion']` but targeted strikes and ground-combat near facilities were being silently ignored. Expanded to `ATTACK_EVENT_TYPES` combining destructive + combat types, and added a dev-only console.debug so the user can inspect event types and facility counts at runtime.

Sections are conditional — facilities without enrichment data just see the existing Water Stress/Aqueduct/Attack sections.

Null-renders when filterStats is absent (cache hits) so the panel doesn't show stale data.
- `waterStore.filterStats` + `setFilterStats` wired end-to-end: server attaches filterStats in non-cached `/api/water` response → `useWaterFetch` forwards via guarded `!== undefined` check → store slot → DevApiStatus selector.
- Bonus consistency fixes bundled: master `showSites` toggle wired into `useEntityLayers.visibleSites` memo, proximity pin filter added to `useWaterLayers` (matches behavior for other entity types).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Added proximity pin filter to water facilities**

- **Found during:** Task 2 (when reviewing useWaterLayers filter sequence)
- **Issue:** Water facilities were the only entity type NOT honoring the user's proximity pin — flights, ships, events, and sites all applied `haversineKm(pin, entity) <= radiusKm`, but water facilities ignored the pin entirely.
- **Fix:** Added proximity filter after the name filter, matches `entityPassesFilters` behavior for other types.
- **Files modified:** src/hooks/useWaterLayers.ts
- **Verification:** grep confirms `proximityPin` used in water layer filter sequence; TypeScript compiles.
- **Committed in:** e335219 (Task 2 commit — bundled with REV-5 since both are in useWaterLayers)

**2. [Rule 2 - Missing Critical] Master `showSites` toggle wired into useEntityLayers**

- **Found during:** Prior session (partial work preserved)
- **Issue:** The filter store has a `showSites` master toggle (analog of `showEvents`) but `useEntityLayers.visibleSites` wasn't consulting it, so toggling the master sites toggle had no effect on the map.
- **Fix:** Added `if (!showSites) return [];` early-out in the visibleSites memo, added dependency to the memo array.
- **Files modified:** src/hooks/useEntityLayers.ts
- **Verification:** Logic matches pattern used for `showEvents` across event filters.
- **Committed in:** 34774b0 (Task 1 commit — bundled as filter consistency work)

**3. [Rule 3 - Blocking] Kept waterTreatment atlas slot instead of shrinking canvas**

- **Found during:** Task 1 (icons.ts edit)
- **Issue:** Plan said "if waterTreatment icon entry exists ONLY for treatment_plant, remove from ICON_MAPPING and the atlas." I removed from ICON_MAPPING but left the atlas canvas drawing code (Icon 15) in place.
- **Fix:** Kept the unused atlas slot at x=480. The canvas is still 576px wide with the slot drawn; no consumer references `waterTreatment` anymore, so the dead pixels are harmless.
- **Rationale:** Removing the atlas shape draw would require renumbering all subsequent icon x-offsets (waterDesalination at 512, triangle at 544) and updating the canvas-width HMR guard. For no functional gain (the slot is invisible), it's not worth the cascade risk. Documented as a decision.
- **Files affected:** src/components/map/layers/icons.ts (left drawing code intact)
- **Committed in:** 34774b0 (Task 1 commit)
