## Accomplishments

- **Two GitHub Actions workflows live at `.github/workflows/`:**
  - `ci.yml` — runs on every PR and every push to main. Three jobs: `lint-and-typecheck` (lint + format:check + typecheck + knip advisory), `test` (vitest --coverage + codecov upload), `audit` (npm audit --audit-level=high). Concurrency group cancels stale PR runs.
  - `codeql.yml` — runs on PR to main + push to main + weekly Monday 06:00 UTC schedule. Uses `javascript-typescript` language matrix with `github/codeql-action/init@v3` then `github/codeql-action/analyze@v3`.
- **Husky v9 + lint-staged v15 installed as devDependencies** with the `prepare: husky` script so `npm install` auto-installs the hooks on fresh clone.
- **`.husky/pre-commit` is a bash script** that (a) runs `npx lint-staged` on staged files only, (b) runs `gitleaks protect --staged --redact --verbose` if the binary is present, (c) prints install instructions and continues if the binary is absent. File permissions are `-rwxr-xr-x`.
- **`.gitleaks.toml` allowlist** covers test fixtures (`server/__tests__/fixtures/**`), planning docs (`.planning/**`), `docs/screenshots/**`, `package-lock.json`, and placeholder patterns like `KEY=your_key_here` / `TOKEN=replace_me`. Uses TOML triple-single-quote literal strings so regex metacharacters don't need double-escaping.
- **`package.json` has a top-level `lint-staged` block** mapping `*.{ts,tsx}` to `["eslint --fix", "prettier --write"]` and `*.{json,md,css,yml,yaml}` to `["prettier --write"]` (package-lock.json skipped via `.prettierignore`).
- **Gitleaks smoke test validated end-to-end:**
  - RED: planted `AKIA1234567890ABCDEF` in `src/__leak-fixture.ts`, staged it, `gitleaks protect --staged --redact --verbose` → exit code `1`, report `leaks found: 1` (rule: `generic-api-key`).
  - GREEN: unstaged and deleted fixture → `gitleaks protect --staged` → exit code `0`, report `no leaks found`.
- **Pre-commit hook measured runtime:** `npx lint-staged` on a single new TypeScript file completed in **~1.5 seconds**, well under the 2-second budget. (Cold first invocation includes eslint startup cost; subsequent commits are faster.)
- **Self-commit proved the hook works end-to-end:** the Task 2 commit (`f355e34`) triggered the pre-commit hook on 4 staged files — lint-staged formatted them via prettier, gitleaks warned about missing binary, commit succeeded.

## Vercel Preview Deploy Enablement (Hand-off to Plan 04)

**Vercel preview deploys are not wired via `.github/workflows/` YAML.** They are enabled through the Vercel ↔ GitHub integration, configured in the Vercel dashboard:

1. Visit `https://vercel.com/<username>/<project>/settings/git`
2. Under **Production Branch**, confirm `main` is selected
3. Under **Git Integration**, confirm the GitHub repo is connected (OAuth app authorized)
4. Under **Deployments**, confirm **Preview Deployments** → **Pull Requests** is toggled ON
5. Optional: under **Environment Variables**, ensure preview env has the same keys as production (so preview deploys don't fail on missing `UPSTASH_*`, etc.)

**Verification:** Open a PR against `main`. Vercel bot should comment on the PR with a preview URL within ~60 seconds of the first push. If no comment appears, check `https://vercel.com/<username>/<project>/deployments` for failed builds.

**Why no YAML:** The Vercel GitHub App already handles preview deployments natively. Adding a `vercel-action` YAML workflow would duplicate this and invalidate the Vercel-managed commit status. Leave it alone.

## Decisions Made

See `key-decisions` in frontmatter. Most consequential:

1. **Task 3 smoke test captured in SUMMARY, not as a permanent package.json script.** The plan explicitly says "NOT as a permanent script" because the point is to prove gitleaks works once, not to keep re-running it locally. CI would catch any regressions via a future `gitleaks-action` invocation (deferred to a later plan).
2. **Pre-commit hook fails open on missing gitleaks binary.** A solo developer on a fresh clone shouldn't be blocked from committing just because they haven't run `brew install gitleaks` yet. The hook prints install instructions and continues. This is documented in Plan 04 README as a "recommended but optional" install step.
3. **`knip` in CI runs with `continue-on-error: true` as advisory only.** Plan 01 wired knip as a cleanup tool, not a gate. During the 26.4 documentation wave we don't want knip warnings to block PRs for pre-existing state. Plan 06 (palantir gaps closure) can tighten this to a hard gate if desired.
