## Decisions Made

- **`redactPaths` array contents and rationale** documented via JSDoc above the export, with 3 categories:
  1. HTTP auth headers (authorization, cookie, x-api-key, set-cookie) — pino-http always logs `req`/`res`, these paths MUST be redacted pre-sink.
  2. Wildcard token fields (`*.UPSTASH_REDIS_REST_TOKEN`, `*.OPENSKY_CLIENT_SECRET`, `*.AISSTREAM_API_KEY`, `*.ADSB_EXCHANGE_API_KEY`) — defensive against adapter logs accidentally capturing process.env slices.
  3. Production-only PII (`req.remoteAddress`, `req.ip`) — conditionally added when `NODE_ENV === 'production'` so dev debugging of bot traffic still works.
- **Type coverage baseline 97.05% (not 99%)**: Pushing above 97 would require cleaning up deck.gl v9 type-cast leaks in `useEntityLayers`, `BaseMap`, `useWaterLayers` — out of scope for 26.4. Gate locked at 97 as a regression ratchet. TODO comment documents the 99% aspirational target.
- **2000ms Redis op timeout**: Upstash healthy RTT is typically 10-80ms, so 2000ms is 25x headroom on the happy path and doesn't affect production latency. But when URL/token is missing or a network partition occurs, the Upstash client retries internally and blocks indefinitely — that was the root cause of the chaos test's 10s-timeout failures before the wrapper. 2000ms caps the worst case and lets the safe wrapper fall through to memCache.
- **Only 3 routes wired to sendValidated (not all 14)**: The plan explicitly scoped this to "3 representative routes" as proof-of-concept. Full rollout across flights, ships, events, news, sites, water/precip, water, markets, weather, sources, geocode, cron-health, cron-warm, health is a mechanical follow-up that can land in a future maintenance pass or alongside Plan 04/05/06 docs work.
- **Sites route needed no changes**: The chaos test initially failed on sites because `cacheGetSafe` was hanging on the Upstash client — not because sites.ts had direct redis calls. Once the timeout wrapper landed in `cache/redis.ts`, sites immediately passed. This is why the expected sites.ts diff (per handoff) turned out to be empty.

**2. [Rule 2 - Missing Critical] Added timeout wrapper to cacheGetSafe / cacheSetSafe**

- **Found during:** Task 3 (chaos test investigation)
- **Issue:** The chaos test mocked `@upstash/redis` at the module level, but `actual.cacheGetSafe` loaded via `importOriginal` retained a closure to the REAL module-scope `redis` const. When the real Upstash client had no URL/token, it logged a warning but still attempted `fetch(undefined)` and retried per its default retry policy — hanging the request indefinitely. Two chaos tests timed out at 10s waiting for events and the aggregate sweep.
- **Fix:** Added `withTimeout()` helper + `REDIS_OP_TIMEOUT_MS = 2000` constant in `server/cache/redis.ts`, wrapped both `cacheGet` and `cacheSet` calls inside the safe variants with `Promise.race`. On timeout, the existing try/catch catches the "timed out" error and falls through to memCache (cacheGetSafe) or swallows it (cacheSetSafe).
- **Files modified:** server/cache/redis.ts
- **Verification:** All 10 chaos tests green in 38s (dominated by 16 timeout waits ~2s each). All 1266 suite tests still green. This is also a real production hardening, not just test scaffolding — a hung Upstash call would have frozen a Vercel lambda until its 60s timeout.
- **Committed in:** e18d683 (Task 3 commit)

**3. [Rule 3 - Blocking] Wrapped events.ts shouldBackfill + backfill writeback in try/catch**

- **Found during:** Task 3 (chaos test investigation, pre-resume)
- **Issue:** events.ts line 45 (`redis.get<number>(BACKFILL_KEY)`) and line 105 (`redis.set(BACKFILL_KEY, ...)`) called the raw Redis client directly with no guard. Under Redis death these would throw and propagate up through the route handler, triggering a 500 via the errorHandler. This is the real Path A gap the handoff flagged.
- **Fix:** `shouldBackfill()` now has its own try/catch that returns `true` on Redis failure (allowing the backfill attempt, which has its own error guard). A new `recordBackfillTimestamp()` helper replaces the inline `redis.set` call and swallows errors (cooldown tracking is non-critical — the worst case is another backfill attempt on the next request, which GDELT rate-limits itself).
- **Files modified:** server/routes/events.ts
- **Verification:** Events chaos test passes (once the timeout wrapper lands alongside this fix). Normal-path events.test.ts also green (18/18 tests).
- **Committed in:** e18d683 (Task 3 commit)
