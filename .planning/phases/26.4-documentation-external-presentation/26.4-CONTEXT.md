  - Remove all `TODO`/`FIXME` comments that aren't the intentional `TODO(26.2)` GDELT-redo flags

### CI/CD (GitHub Actions)

- **Workflow 1: `.github/workflows/ci.yml` — runs on every PR**
  - `npx tsc -b` (strict mode across server + app)
  - `npx vitest run` (full 1241-test suite)
  - `npm audit --audit-level=high` (security scan)
  - GitHub CodeQL analysis (security scan, free for public repos)
  - Coverage upload to Codecov (or Coveralls) → renders README badge
- **Workflow 2: Vercel preview deploy on PR** — enabled via Vercel ↔ GitHub integration, not custom YAML
- **Type coverage gate:** CI runs `type-coverage` and fails if below 99% (captured in Palantir Gaps section below)

### Pre-commit Hooks (husky + lint-staged)

- **Scope:** fast only — ESLint + Prettier on staged files
- **Target runtime:** <2s per commit — never slow the workflow
- Skip tsc and test runs in pre-commit (CI catches those on push)
- **Exception:** gitleaks pre-commit hook scans staged files for leaked secrets (API keys, tokens) — cheap paranoia, high value for an OSINT tool

### Palantir-Grade Gaps We Missed in 26.3

These close specific engineering gaps that 26.3 didn't address:

1. **Log redaction for secrets & PII**
   - Pino `redact` config to scrub: API key headers (authorization, x-api-key), Upstash tokens, client IPs (in prod), upstream auth payloads
   - Add test that verifies known sensitive field paths never reach the transport output
   - Critical because this is an OSINT tool handling geolocation data

2. **Type coverage measurement + CI threshold**
   - Install `type-coverage` package
   - Add CI gate at 99% (any `any` escape hatch fails the build)
   - Surface the metric in README Engineering section
   - This is a unique portfolio metric — proves strict mode bought real safety

3. **Chaos test: graceful degradation when Redis dies**
   - End-to-end test that simulates Upstash REST failure (mock throws)
   - Verifies: in-memory fallback path activates, stale cache responses return with `stale: true` flag, /health endpoint reports degraded state, no 500s to the client
   - Proves the resilience claims we make in docs

4. **Response schema validation (Zod at output)**
   - Parse API responses through Zod schemas BEFORE sending to client, not just incoming queries
   - Catches drift between implementation and OpenAPI spec
   - Fail loudly in dev (500 + structured log), fail open in prod (log warning, send anyway to avoid cascade)
   - Belt-and-suspenders Palantir-style validation at both I/O boundaries

## Specific Ideas

- **Hormuz GIF framing:** the GIF should start zoomed out showing the broader Middle East, then pan/zoom to the Strait of Hormuz with all layers active — demonstrating the "what's happening around a specific chokepoint" workflow that's the tool's actual value. Shows political layer (Iran red, Saudi/UAE blue), ethnic layer (Arab/Persian boundaries), water stress (Gulf salinity/facilities), threat clusters (if any recent events), oil markets reacting. This is the pitch in motion.
- **Engineering section:** model after repos like `radix-ui/primitives` or `tanstack/query` — badges at the top, metrics front and center, proof-of-discipline visible without scrolling.
- **"What I'd do differently" section:** explicitly name Phase 26.2 (the NLP approach that was scrapped) as an honest retrospective. This is a high-signal artifact: showing you can kill your darlings is more portfolio-valuable than pretending everything went smoothly.
- **As-built honesty:** the diagrams should label known tech debt, not hide it. `TODO(26.2)` hardcoded tables explicitly annotated. Reviewers spotting undisclosed warts is worse than disclosed warts.
- **Runbook style:** one failure mode per section, with symptom, cause, remediation, detection. Model on Google SRE book runbook patterns.
