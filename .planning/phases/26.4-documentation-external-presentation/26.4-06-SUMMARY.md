  `stale: true` envelope semantics, the yellow StatusPanel dot,

  `sendValidated<S>` dev-throw / prod-warn divergence and the
  `.passthrough()` drift tolerance on nested data fields.
- **Frontend Layer: Lost Contact** — documents `useSelectedEntity`
  ref-based last-known tracking with the grayscale banner.
- **Rate Limit Layer** — documents the opposite-of-cache
  fail-open behavior (let traffic through when the counter is
  down, rather than block legitimate users).
- **`/health` Endpoint Contract** — documents the stable response
  shape, the HTTP-200-always policy, and the consumer list.
- **Summary table** at the bottom mapping each layer to failure
  mode, user-facing outcome, and the test/code-path proof.

## Decisions Made

See `key-decisions` in frontmatter. The most consequential:

1. **ADR-0005 written at 300 lines, 5× the minimum.** The user
   explicitly flagged this as the highest-portfolio-signal artifact
   in the phase. Phoning it in at 60 lines would have missed the
   point. The extra depth delivers the specific files deleted, the
   revert commits, the 2-week time investment honestly, a 4-lesson
   "What I Learned" section with rules for next time, and a
   forward-looking "what to do instead" section naming the leading
   redo candidate (upstream `NumSources` + full noisy-CAMEO
   filter). This ADR is the difference between "documented a
   scrap" and "demonstrated judgment by documenting a scrap."
2. **Runbook failure modes grounded in real code paths, not generic
   SRE advice.** Every Detection / Remediation / Prevention bullet
   points at a specific file and often a specific line range.
   "`cacheGetSafe` + new `REDIS_OP_TIMEOUT_MS` timeout wrapper at
   lines 19-42 of `server/cache/redis.ts`" is a reviewable claim,
   not hand-waving. This is the as-built honesty principle from
   Plan 05 carried through to the runbook.
3. **ADR-0001 Consequences section explicitly names the 2000 ms
   timeout as production hardening.** Rather than writing a new
   ADR-0009 specifically for the chaos-test-driven fix, the
   hardening is integrated into the existing Upstash ADR's
   Negative consequences section. Keeps the decision record
   coherent — "we chose Upstash, and here's how the choice has
   evolved" is cleaner than "we chose Upstash (ADR-0001) and
   separately we added a timeout (ADR-0009)."
4. **degradation.md uses a proof column in the summary table.**
   Every contract maps to either a test file or a specific code
   line. This is the belt-and-suspenders version of "we document
   our resilience" — the documentation is only believable if it
   points at a proof.
5. **README edits are additive and scoped.** The hero GIF, quick
   start, test matrix, and data sources table are all untouched.
   Only three scoped edits (Architecture links, new Engineering
   Documentation subsection, retrospective blockquote). This
   protects the existing Plan 04 README structure from accidental
   regression.
