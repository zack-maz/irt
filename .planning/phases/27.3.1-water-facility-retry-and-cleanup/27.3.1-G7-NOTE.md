# Phase 27.3.1 G7 — "No water facilities showing DESTROYED" closeout

**UAT Test 9 report:** "not seeing any destroyed" — the user observed
zero water facilities on the map rendering the attacked (dark purple)
styling despite active GDELT conflict events.

**Resolution:** NOT a code defect. Data-reality finding confirmed by
27.3.1-DIAGNOSIS.md §G7.

## Evidence

1. **/api/events is populated.** 168 events in the current window
   including 66 attack-type (31 airstrike, 27 on_ground, 6 explosion,
   2 targeted). Rules out hypothesis C (stale data).

2. **Global minimum attack→facility distance is 6.44 km.**
   - Nearest pair: Jerusalem airstrike → Beit Zait Reservoir (6.44 km)
   - Next: Qargah Dam (14.11 km)
   - Third: Kuwait reservoir (18.01 km)

   The attack-radius threshold hard-coded at three sites:
   - `src/hooks/useWaterLayers.ts:194` — `ATTACK_RADIUS_KM = 5`
   - `src/components/detail/WaterFacilityDetail.tsx:49` — same
   - `src/components/counters/useCounterData.ts:324` — same

   Zero pairs within 5 km means zero destroyed facilities. This is
   CORRECT behavior given the threshold + the live event data.

3. **Shared constant wiring intact.** All three consumers import
   `WATER_ATTACK_EVENT_TYPES` from `src/lib/waterAttackEvents.ts`
   (established Phase 27.3 Plan 03 / WR-01):
   - useWaterLayers.ts:9
   - WaterFacilityDetail.tsx:5
   - useCounterData.ts:13

   Rules out hypothesis B (Plan 06 broke the cross-reference).

4. **Plan 06 scope proof.** Phase 27.3.1 Plan 06 (commit 9eb1465 —
   computeAdmissionDecision extraction) touched only:
   - `server/adapters/overpass-water.ts`
   - `server/__tests__/adapters/overpass-water.test.ts`

   Zero touches to `src/hooks/`, `src/lib/waterAttackEvents.ts`,
   `src/components/detail/`, or `src/components/counters/`. No
   Phase 27.3.1 commit modified any attack-status code path.

5. **Haversine units verified km throughout.** All three consumer
   files use the same `ATTACK_RADIUS_KM = 5` constant compared
   against a haversine distance in kilometres. No unit mismatch.

## Disposition

**Closed as resolved — no code change required.**

The behavior is correct. Zero facilities are destroyed because no
current attack event is within 5 km of any admitted water facility.
The 6.44 km Jerusalem/Beit-Zait near-miss is noted for historical
reference.

## Optional future follow-ups (NOT in 27.3.1)

If the user later requests broader destroyed-water visibility, two
tuning options exist. Neither is a bug fix; both would be discretionary
UX decisions requiring their own discussion:

1. **Widen ATTACK_RADIUS_KM from 5 → 10 km.** Would surface Beit Zait
   - Qargah + 1 Kuwait reservoir as destroyed today. A 3-line change
     across the three consumer files. Tuning decision, not a bug fix.

2. **Add a dev-only "nearest attack: X km" badge in
   WaterFacilityDetail.** Adds transparency — the user would see
   "Beit Zait Reservoir — nearest attack: 6.44 km" and understand
   why it is not flagged destroyed. Implementation: compute haversine
   min distance in the detail component; only render when distance
   < (ATTACK_RADIUS_KM × 2) to avoid cluttering all detail panels.

Both are Phase 28+ candidates if the threshold debate resurfaces.
