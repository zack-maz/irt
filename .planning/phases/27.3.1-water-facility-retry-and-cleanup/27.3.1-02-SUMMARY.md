# Phase 27.3.1 Plan 02: Officially-Recognized-With-A-Name Admission Gate — COMPLETE

## One-liner

Enforced the "every admitted water facility has a real OSM name" directive via a
three-stage compound admission gate (D-05 hasName mandatory → D-06 OR-of-signals →
D-08 score floor secondary), closing the unnamed-admission hole in priority countries
and retiring Round 3 Package A's bare-name-in-non-priority relaxation.

## What shipped

### New export: `hasCapacityData(tags)`

```typescript
export function hasCapacityData(tags: Record<string, string>): boolean {
  return !!(
    tags['height']?.trim() ||
    tags['volume']?.trim() ||
    tags['capacity']?.trim() ||
    tags['area']?.trim()
  );
}
```

Six unit tests cover all four keys + both negative cases (missing keys, empty-string
values). Exported for Plan 06's planned `computeAdmissionDecision` consolidation to
build on.

### Admission gate: structural before/after

**Before (Round 3 / Package A, three scattered branches):**

```typescript
// REV-2 reservoir-only branch
if (facilityType === 'reservoir') {
  if (!(isNotable(el.tags) || hasName(el.tags))) {
    rejections.not_notable++;
    return null;
  }
}
// Dam-only no-priority-no-name
if (facilityType === 'dam' && !inPriority && !hasName(el.tags)) {
  rejections.no_name++;
  return null;
}
// Score floor
if (score < MIN_NOTABILITY_SCORE) {
  rejections.low_score++;
  return null;
}
```

**After (D-05/D-06/D-07/D-08 unified):**

```typescript
// D-05 + D-07: hasName mandatory for ALL types, ALL countries
if (!hasName(el.tags)) {
  rejections.no_name++;
  return null;
}
// D-06: compound gate — hasName AND (isNotable || inPriority || hasCapacityData)
if (!(isNotable(el.tags) || inPriority || hasCapacityData(el.tags))) {
  rejections.not_notable++;
  return null;
}
// D-08: MIN_NOTABILITY_SCORE demoted to secondary floor
if (score < MIN_NOTABILITY_SCORE) {
  rejections.low_score++;
  return null;
}
```

### D-10 resolution: KEEP the GENERIC_TYPE_RE sentinel

The fallback path in `getWaterFacilityDisplayName` is still reachable even after D-05
hardens the admission gate. Reason:

- `hasName(tags)` returns true for non-Latin-only names.
- `extractLabel` (overpass-water.ts:371-379) has an `isLatin` guard that drops
  non-Latin-only names for display, falling back to `FACILITY_TYPE_LABELS[facilityType]`.
- A Persian- or Arabic-only named facility therefore admits via D-05 and arrives at
  `getWaterFacilityDisplayName` with `label = 'Dam'` (or 'Reservoir' / 'Desalination Plant').

**Decision: keep the fallback + document the reach path + add regression test.** The
inline comment above `GENERIC_TYPE_RE` now names the non-Latin-only case explicitly and
stamps the audit date. A new regression test ("D-10 audit: non-Latin-only OSM name path
produces bare-type label, sentinel catches it") proves the fallback chain handles the
case correctly by falling through to the coordinate fallback.

## Expected effect on the Plan 01 baseline

Plan 01 refresh log (`27.3.1-R01-REFRESH-LOG.md`) recorded: dams=4,588 /
reservoirs=892 / desalination=63. Plan 02 changes the following admission behavior:

- **Unnamed dams in priority countries** — previously admitted via the score floor (score ≥
  15 via priority bonus alone). Now rejected via D-05 → expect dams to drop substantially.
  The Plan 01 log's first sample (`"label": "Dam near Afghanistan"` — no name) is the
  canonical rejection target.
- **Unnamed reservoirs in priority countries** — same D-05 rejection, but pre-R-03 they
  were already caught by REV-2 (`isNotable || hasName`). Minimal delta expected.
- **Bare-named reservoirs in non-priority countries** (Pakistan, Oman, Sudan) — Round 3
  Package A admitted these on hasName alone; D-06 now requires a second signal. Expect
  reservoir drop outside the 14-country priority set.
- **Desalination** — all 63 raw entries carry `man_made=desalination_plant` and one or
  more of name/operator (100% admit rate). Plan 02 does not tighten desalination
  admission; Plan 04 may add a capacity floor per CONTEXT.md R-02/D-03.

Plan 04 (R-02 calibration) will run the first refresh after this change and capture the
new numbers.
