## Decisions Made

- **hasName tightening scope:** removed isNotable shortcut but RETAINED isNotable as a standalone function used inside the D-06 compound gate. Decoupling "admission signal" from "display-Latin-script?" — hasName tells us whether to ADMIT the facility, extractLabel tells us how to DISPLAY it. These are orthogonal concerns now.
- **Pakistan out of scope:** DIAGNOSIS §G2 floated adding Pakistan as a parallel expansion. Plan 10 frontmatter + objective section explicitly out-of-scoped it because UAT evidence was strictly about Turkey over-representation (G2), not Pakistan under-representation. If post-regen review shows Pakistan coverage is thin, a separate backlog item (not Plan 10) opens.
- **Decision order — excluded_turkey AFTER excluded_location:** the sites adapter uses a single bbox filter for Turkey; water adapter needed per-coordinate logic because the 600km-from-Diyarbakir rule ALREADY handles western/central Turkey (the bulk of Turkish infrastructure). `excluded_turkey` catches only the eastern/central SE-Turkey band that was the Kurdish-conflict-zone carve-out — keeping the two buckets semantically distinct makes DevApiStatus readable (geographic-rule vs country-attribution).
- **waterLabeling.ts full deletion over soft-disable:** considered adding a feature flag to keep the labeler but never fire it. Rejected — the labeler's ONLY path produced `"Dam near X"` labels that the user explicitly called out as unwanted in UAT G1. Retaining it as dead code would only create resurrection temptation.
- **Bare-type labels on non-Latin name admits accepted:** 137 of 436 facilities carry a bare `"Dam"` / `"Reservoir"` label because their OSM name is Persian/Arabic/Hebrew only. Client `getWaterFacilityDisplayName` recognizes these via GENERIC_TYPE_RE and renders `"Dam at 33.2°N 44.4°E"` instead. Server-side is the source of truth for admission; client is responsible for display fallback.
