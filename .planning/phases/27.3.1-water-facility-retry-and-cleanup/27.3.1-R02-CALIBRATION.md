# R-02 Calibration Log

Records the per-type count calibration for water facility admission against
the user's target bands (D-01 dams 300-500, D-02 reservoirs 300-500,
D-03 desalination 10-25). Wave 2 of Phase 27.3.1, executed after the
Wave 1 admission-gate hardening (Plan 02 / R-03) and observability
expansion (Plan 03 / R-08) shipped.

The calibration philosophy from CONTEXT.md D-04 is the policy under which
this log operates:

- Target is the ceiling, not the floor — UNDER-pulling is fine if every
  facility has a real name + notability signal.
- Err high over low (user quote).
- Tune with `filterStats` visibility (per-type rejection bucket deltas),
  not blind threshold changes.
- Don't tune to hit the center of the band — a clean admission at 320 is
  better than a messy 420.

## Decision

**Branch 2b-(ii) + Branch 2c, applied as a single coupled change.**

The Plan 04 decision tree allowed at most ONE code change per iteration.
The dam/reservoir over-pull and the desalination under-pull are coupled:
tightening the compound gate without exempting desalination would drop
desal further below floor (already 4 short). Treating them as a single
atomic change is the only way to land all three types in band in one
iteration.

### Branch 2b-(ii) — tighten compound to 2-of-3 (dams + reservoirs)

> Replace `passesCompound = isNotable || inPriority || hasCapacityData`
> with a count: `signalCount = [isNotable, inPriority, hasCapacityData].filter(Boolean).length;
if (signalCount < 2) reject as not_notable`.

Rationale (citing Run 1 evidence):

- byCountry shows Turkey (509), Saudi Arabia (262), Iran (234), Iraq
  (177), UAE (175) dominating. These are all priority. Most facilities
  in those countries are name-only with no wikidata and no capacity
  tags. Single-OR + priority = floodgate. 2-of-3 closes it.
- scoreHistogram bucket 25-39 holds 1471 facilities — those are the
  name-only-in-priority entries. A 2-of-3 gate slices all of them; the
  remaining 681 (40+ score) preserve the high-confidence admit set.
- byTypeRejections shows `not_notable: 302` (dams) + `305`
  (reservoirs) — the OR gate is mostly idle pre-tightening. After
  tightening, this bucket grows by an order of magnitude (Run 2 below
  shows 1103 / 1063), proving the gate is now doing work.

Plan 04 Step 2 explicitly preferred (i) `MIN_NOTABILITY_SCORE` bumps
over (ii) compound restructuring because it has a smaller change
surface. Why (ii) was chosen instead:

- `MIN_NOTABILITY_SCORE` is currently 15. To slice the 25-39 bucket,
  it would need to climb to ≥40. That bump would also drop the entire
  `40-54` bucket of 101 facilities (mixed-quality) without offering
  per-type control.
- Critically, `MIN_NOTABILITY_SCORE` would also slice desalination
  (which has a +5 type bonus → name-only desal in non-priority scores
  20). Bumping the floor to 40 would drop desal to ~3, deepening the
  under-pull. Branch (ii) preserves per-type granularity.
- `MIN_NOTABILITY_SCORE` is a coarse opaque score; the compound gate
  speaks in named, debuggable signals. The user-facing CALIBRATION
  evidence (this document) is more reviewable when the rule is
  expressed in compound semantics.

### Branch 2c — desalination exemption

> Add an explicit desalination exemption to the D-06 compound: hasName=true
> on facilityType==='desalination' admits regardless of compound result.

Rationale:

- Run 1 desal admit set was 6 of 63 raw. Under 2-of-3 compound, three
  of those 6 (the score-20 entries with name only) would drop further,
  landing the count at ~3 — well below the 10-25 floor.
- Desal OSM coverage in the bbox is sparse (63 raw across the entire
  ME). The name+desalination-type combination carries enough notability
  signal on its own per the user's "10-25" target. Strict 2-of-3 over-
  filters this small population.
- The +5 desalination bonus in `computeNotabilityScore` already
  acknowledges this asymmetry. The exemption is the natural extension
  of that decision into the compound gate.
- D-05 hasName is preserved (test
  `R-02 desalination exemption: still rejects unnamed desalination`
  proves it). Unnamed desal still rejects.

### Why no second iteration

Run 2 (below) lands desalination in band (15) and dams within 3% of
the ceiling (515 vs 500). Reservoirs at 73 are below the 300 floor —
~227 short. Per the calibration philosophy:

> "Target is the ceiling, not the floor — UNDER-pulling is fine if every
> facility has a real name + notability signal."

> User quote: "Willing to go below 400 for those entities."

A second iteration to climb reservoirs from 73 toward 300 would require
relaxing the gate (e.g. allowing 1-of-2 with named + capacity-or-priority).
That would also bring back the long-tail admits the calibration just
trimmed, sacrificing quality for count. The user's directive is unambiguous:
"officially recognized with a name" + a second notability signal. 73
reservoirs that all clear that bar is a higher-fidelity admission set
than 320 reservoirs that mostly don't.

`outcome: single_tune` (Plan 04 frontmatter).

## Observations Plan 05/06 should know

1. **Scoring system effectively idle.** `low_score` rejection bucket is
   zero pre-AND-post calibration. Plan 06 cleanup (D-22) was already
   considering whether to drop scoreHistogram if it's not driving
   decisions; this calibration confirms `MIN_NOTABILITY_SCORE` is
   redundant with the compound gate. Plan 06 could collapse the
   secondary floor entirely or keep it as a defense-in-depth
   regression sentinel.

2. **Reservoirs at 73 is the band's edge — snapshot stability matters.**
   Run 2's reservoir count of 73 is well below the 300 floor. If a
   future Overpass refresh adds/removes a handful of named-with-capacity
   reservoirs, the count could swing 30-40% in either direction. Plan 05's
   committed JSON snapshot should freeze the current admit set so the
   number doesn't drift; the snapshot's `generatedAt` timestamp is the
   ground truth, not a live re-fetch.

3. **Cyprus desal cluster previously hidden.** 4 Cyprus plants surfaced
   under R-02 (Famagusta, Limassol, Dhekelia, Potima Bay) that were
   not in the Run 1 admit set. Worth a sanity check that Cyprus
   desalination coverage is intentional (Cyprus is non-priority and
   geographically EU-adjacent — consistent with the user's water layer
   bbox of "Greater Middle East").

4. **3 desal entries fall through to "Desalination Plant" generic
   label.** Same pattern as Plan 02 D-10 audit found — non-Latin OSM
   names hit `extractLabel`'s isLatin guard and fall back to the bare
   type label. Plan 05 (snapshot) generation should run `labelUnnamedFacilities`
   so the snapshot bakes in the reverse-geocoded "Desalination Plant
   near {city}" labels at refresh time.

5. **R-02 changes the desalination contract.** Pre-R-02, named desal
   admission required `isNotable || isPriorityCountry || hasCapacityData`.
   Post-R-02, only hasName matters. If Plan 06 consolidates the gate
   into `computeAdmissionDecision`, the desalination exemption MUST
   be preserved as an explicit per-type branch — losing it would
   re-collapse desal to ~6.
