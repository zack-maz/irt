Wired a committed pre-enriched water facility snapshot (`src/data/water-facilities.json`) as the cold-start floor between the dev file cache and Overpass, plus a manual `npm run refresh:water` script that regenerates the file end-to-end (Overpass → classify → 2-of-3 compound gate → enrich → reverse-geocode → atomic write). ## Tier Structure (R-07 invariant)

```
  ┌──────────────────┐
  │ GET /api/water   │
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────────────────────────────┐
  │ Tier 1: Redis (water:facilities, 24h TTL)│  ← canonical on user requests
  └────────┬─────────────────────────────────┘
           │ miss
           ▼
  ┌────────────────────────────────────────────────┐
  │ Tier 2: Dev file cache (.dev-cache/, dev only) │  ← gitignored, ephemeral
  └────────┬───────────────────────────────────────┘
           │ miss (always in production)
           ▼
  ┌──────────────────────────────────────────────────────────┐
  │ Tier 3: Snapshot (src/data/water-facilities.json, R-04)  │  ← committed, cold-start floor
  └────────┬─────────────────────────────────────────────────┘
           │ miss (snapshot absent OR malformed)
           ▼
  ┌──────────────────────────────────────────────────────────┐
  │ Tier 4: Overpass (refresh gate: dev env OR vercel-cron)  │  ← never on user path in prod
  └──────────────────────────────────────────────────────────┘
```

In production (NODE_ENV=production, no vercel-cron UA): Tier 2 always misses, Tier 4 is gated off for user requests. The snapshot is the guaranteed cold-start fallback. `npm run refresh:water` is the only developer-facing path to Tier 4 — and even then it's explicitly an offline pipeline, not a runtime path.

```bash
# refresh=true bypasses both Redis and snapshot
$ curl -sS http://localhost:3001/api/water?refresh=true | jq -r '.filterStats.source, (.filterStats.overpass | length)'
overpass

```

The Overpass array from the refresh response:

```
  dams         primary  status=200 18089ms ok=true
  reservoirs   primary  status=200 31736ms ok=true
  desalination primary  status=429 7917ms  ok=false  ← rate limited, fallback fires
  desalination fallback status=200 6084ms  ok=true
```

This is the exact same failure mode the refresh script encountered — the primary Overpass instance rate-limits the desalination query at certain times; the fallback mirror recovers. This rate-limit behavior is PART of the R-04 motivation (another reason to take Overpass off the request path) and confirms the mirror retry logic still works.

## Notes for Plan 07 (R-05 sites parity)

Plan 07 will mirror this exact structure for sites. Here are the decisions/observations Plan 07 should carry over:

1. **Atomicity matters.** Use tempfile + rename in `scripts/refresh-sites.ts`. Sites will be larger than water (more OSM elements per-country), and a crashed refresh mid-write could produce a file large enough that a bad partial parse takes a while to detect.

2. **Deterministic ordering.** Sort `features[]` by OSM id (or equivalent stable key). Round coords to 6dp. Pretty-print with 2-space indent + EOF newline. These are the conventions `src/data/rivers.json` / `src/data/aqueduct-basins.json` / `src/data/water-facilities.json` all follow.

3. **Share the labeler.** If sites has its own unnamed-fallback logic (I don't think it does currently, but Plan 02 added one for water), extract it to `server/lib/` and share between route + script. Don't re-implement.

4. **Mock the snapshot loader in resilience tests.** If Plan 07 adds `loadSitesSnapshot`, preemptively mock it in `server/__tests__/resilience/redis-death.test.ts`. Without the mock, 24h-stale snapshots + dead Redis = labelUnnamedFacilities-style timeouts.

5. **Source tier override in the loader.** Snapshot loader forces `source='snapshot'`. Plan 07's loader should do the same — defensive layering keeps provenance accurate no matter what the refresh script persisted.

6. **refresh=true still bypasses snapshot.** The existing dev-or-cron gate on Overpass stays. Don't accidentally close it off in the process of wiring the snapshot tier.

7. **The Redis-populate-from-snapshot pattern works cleanly.** First request loads snapshot, calls `cacheSetSafe(FACILITIES_KEY, data, TTL)`, returns `source='snapshot'`. Second request hits warm Redis, returns `source='redis'`. Plan 07 replicates this with sites:\* keys.
