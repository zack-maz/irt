## Decisions Made

- **Option A "cache the full envelope" over runtime shape-guard code** per DIAGNOSIS §G3+G4 and threat model T-27.3.1.11-04. Runtime shape guards (`if (Array.isArray(cached.data))`) would add per-request overhead and an accumulation of branch code; key bump forces atomic cutover at deploy time with zero runtime cost.
- **source=redis override on cache-hit is a deliberate UX fix.** The cached filterStats was originally populated by the snapshot tier (first warm write) or the Overpass tier (refresh); rendering `source: 'snapshot'` on every subsequent cache-hit would be misleading because DevApiStatus provenance reports "where did THIS response come from" (the serve tier), not "where did the stats originate" (the warm tier). Documented in both water.ts and sites.ts cache-hit branch JSDoc.
- **Dev file cache still reports source=snapshot, NOT source=redis** — the dev file cache is semantically the cold-start floor (same tier as the snapshot), just ephemeral/gitignored instead of committed. DevApiStatus provenance honesty matters more than branch uniformity here.
- **buildEmptyFilterStats retained but no longer invoked on happy paths** — in water.ts it's still called in the error-without-cache fallback (legitimate — no real stats exist because fetchWaterFacilities threw before returning). In sites.ts it's no longer called anywhere (the 502 path has no body), but kept for Plan 12+ future defensive paths rather than re-implementing the shape later. Silenced with `void buildEmptyFilterStats` in sites.ts to avoid unused-export warnings.
- **Test fixtures use `fetchedAt: Date.now() - 1000` for fresh cache-hit tests.** Initially one test (`cache-hit response overrides generatedAt to cache lastFresh ISO`) used a fixed constant `1_700_000_000_000` (Nov 2023 — >24h ago as of 2026-04-19), which made the cache entry stale and routed through the snapshot branch instead of the cache-hit branch. Rule 1 fix caught during GREEN verification — all cache-hit tests now use `Date.now()` or `Date.now() - 1000` for recency.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Test fixture used historical timestamp for "fresh" cache-hit test**

- **Found during:** Task 1 GREEN verification (`npx vitest run server/__tests__/routes/water.test.ts`) — 20/21 pass, 1 failing.
- **Issue:** The test `cache-hit response overrides generatedAt to cache lastFresh ISO` used `const lastFreshMs = 1_700_000_000_000` (Nov 14, 2023) as the cache's `fetchedAt`. Relative to the test runtime (2026-04-19), that's >24h ago, so `cached.stale === true` in the route's cache-hit check and the route falls through to the snapshot tier instead of the cache-hit branch. The test was actually exercising the wrong branch — the assertion checked a `generatedAt` value that only the cache-hit branch writes.
- **Fix:** Changed to `const lastFreshMs = Date.now() - 1000` (1 second ago — guaranteed fresh within the 24h logical TTL).
- **Files modified:** `server/__tests__/routes/water.test.ts` (1 line).
- **Verification:** All 21 water tests pass; the assertion now verifies the cache-hit branch's generatedAt override as intended.
- **Committed in:** `aaad081` (Task 1 GREEN commit, bundled with water.ts implementation).

**2. [Rule 2 — Missing Critical] Dev file cache write wraps the envelope with best-effort stats fallback**

- **Found during:** Task 1 Step 3 Site A (`server/routes/water.ts` dev file cache branch).
- **Issue:** The plan's action text for Site A specified wrapping the dev cache's `{facilities, stats}` shape in an envelope when writing to Redis, but the dev cache's `stats` field is typed `unknown` (not `WaterFilterStats`) because the cache file format predates strict typing. A naïve wrap would make Redis carry `unknown` as filterStats, failing the `<WaterCachePayload>` type contract and tripping schema validation on the next cache-hit read.
- **Fix:** Added defensive cast + fallback: `const devFilterStats: WaterFilterStats = (devCached.stats as WaterFilterStats | undefined) ?? buildEmptyFilterStats('snapshot', new Date().toISOString())`. If the dev cache file predates Plan 03's `{facilities, stats}` format and has no stats, the envelope still gets a well-formed (zero-tally) filterStats object so Redis never stores malformed data.
- **Files modified:** `server/routes/water.ts` (6 lines added in dev file cache branch).
- **Verification:** Typecheck clean; dev file cache branch test flow remains unbroken (existing test mock `loadDevWaterCache → null` falls through this branch, so no new test coverage needed — but the branch would now survive a real dev cache file with missing stats).
- **Committed in:** `aaad081` (Task 1 GREEN commit).

**3. [Rule 2 — Missing Critical] `/api/water/precip` miss-path also writes the envelope**

- **Found during:** Task 1 Step 3 Site D (`/api/water/precip` facilities-load branch).
- **Issue:** The plan specified that the precip handler's miss-path fetch-and-cache should store the envelope, but the pre-Plan-11 code discarded the `stats` field from `fetchWaterFacilities()` on this path (`const { facilities: freshFacilities } = await fetchWaterFacilities();`). A naïve envelope write would need to either re-fetch stats or fabricate an empty one — either way Redis would still warm with incomplete observability for the first subsequent `/api/water` hit.
- **Fix:** Destructured `stats: freshStats` from the fetch and passed it into the envelope: `await cacheSetSafe(FACILITIES_KEY, { facilities, filterStats: freshStats }, WATER_REDIS_TTL_SEC);`. This closes a subtle observability leak — if a precip request warms Redis before the main /api/water request, the observability tallies are now populated immediately instead of zeroed until the next Overpass refresh.
- **Files modified:** `server/routes/water.ts` (/api/water/precip branch, ~10 lines).
- **Verification:** All 21 water tests pass including the new `/api/water/precip unwraps .facilities from cached envelope` test.
- **Committed in:** `aaad081` (Task 1 GREEN commit).

**4. [Rule 2 — Missing Critical] Error-with-cache branches spread cached filterStats (both routes)**

- **Found during:** Task 1 Step 4 Site B (water.ts error branch) and Task 2 Step 4 Site B (sites.ts error branch).
- **Issue:** The plan specified updating the main cache-hit branches to spread `cached.data.filterStats` but didn't explicitly call out the error-with-cache fallback branches (the branch that serves a stale cached response when `fetchWaterFacilities` / `fetchSites` throws). Pre-Plan-11 those branches used `buildEmptyFilterStats('redis', ...)` — which means under degraded serve, R-05/R-08 observability would still be zeroed even though the cached payload has real stats. That defeats half the point of the envelope.
- **Fix:** Both error-with-cache branches now also spread `cached.data.filterStats` with the same source-override pattern as the main cache-hit branch. Added new test in each file: `error path with stale cached payload returns stats from cache with source=redis`.
- **Files modified:** `server/routes/water.ts` (~10 lines), `server/routes/sites.ts` (~10 lines).
- **Verification:** New tests pass in both files; chaos test's error-path coverage unaffected.
- **Committed in:** `aaad081` (water.ts branch) + `d9afdda` (sites.ts branch).

**5. [Rule 2 — Missing Critical] Silence unused buildEmptyFilterStats in sites.ts**
