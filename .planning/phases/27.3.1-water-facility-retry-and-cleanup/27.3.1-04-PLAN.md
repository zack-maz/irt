<objective>
Calibrate the water admission gates so per-type counts land in the user's target bands:
D-01 dams ~400 (300-500), D-02 reservoirs ~400 (300-500), D-03 desalination 10-25. Tune
only when R-01 evidence + R-08 per-type rejection breakdowns show a specific gate is
starving or flooding a category. Commit an evidence trail.

Purpose: The Package A changes (debug round 3) were based on a hypothesis. R-03 Plan 02
tightened the gate structurally — but the actual counts depend on real OSM data. This
plan closes the loop: reads the post-R-03 + post-R-08 refresh numbers, decides whether
to nudge MIN_NOTABILITY_SCORE (D-08 secondary gate) or reshape the D-06 compound formula,
and records the decision.

Output:

- Post-R-03 refresh log showing current counts
- If any type is out of band: targeted threshold change + re-verification
- `27.3.1-R02-CALIBRATION.md` documenting the tuning process
- At most ONE code change to server/adapters/overpass-water.ts (either MIN_NOTABILITY_SCORE
  value or a specific D-06 compound tweak) — prefer doing nothing if counts are already
  in-band

Scope: Pure tuning. No new gates, no structural changes. If the post-R-03 refresh shows
all three types already in-band, this plan becomes a documentation pass and the
"code change" task is skipped.
</objective>

    **Step 1 — Classify each type's state.**

    Given the post-R-03 filteredCounts from Task 1, label each type:
    - `in_band`: within the target range (dams 300-500, reservoirs 300-500, desalination 10-25)
    - `below`: lower bound of the band (need looser gate)
    - `above`: upper bound of the band (need tighter gate)

    Record the classification in the CALIBRATION.md file (Step 4). From here the decision
    tree branches based on which types are out-of-band.

    **Step 2 — Decide the adjustment (at most ONE code change).**

    Branch 2a — **all three types in_band** → no code change. Skip to Step 4.

    Branch 2b — **reservoirs or dams above band**. The D-06 compound gate has three branches:
    isNotable (wikidata/wiki), isPriorityCountry, hasCapacityData. If one country dominates
    byCountry (e.g. Iran: 600 reservoirs), the priority branch is the flood source. Tuning
    options (pick one):

    (i) **Raise MIN_NOTABILITY_SCORE** from 15 → 20 (or 25 if delta suggests). This trims
        the long tail of name-only facilities in priority countries. Evidence required:
        scoreHistogram shows many entries in the 15-25 bucket.

    (ii) **Tighten D-06** by requiring TWO of the three compound signals rather than one.
         Example: replace `const passesCompound = isNotable || inPriority || hasCapacityData`
         with a count: `const signalCount = [isNotable, inPriority, hasCapacityData].filter(Boolean).length; if (signalCount < 2) { rejections.not_notable++; ... }`. Heavier
         change — use only if (i) fails to bring counts into band.

    Prefer (i) — smaller change, regression surface is one constant.

    Branch 2c — **desalination below band**. The desal-specific +5 bonus already helps.
    The band target is 10-25 and pre-Package-A pulled 13, so the expected range is near
    the low edge. If count < 10:
    - Check byTypeRejections.desalination. If low_score dominates, relax MIN_NOTABILITY_SCORE
      back toward 10 (but this would also flood dams/reservoirs — avoid).
    - Better: add an explicit desalination exemption to the D-06 compound — named desal in
      ANY country admits (desal is rare enough that any named plant is likely notable).
      ```typescript
      // D-06 per-type exemption: named desalination plants admit anywhere in the bbox.
      // Desal OSM coverage is sparse (~60 raw elements across ME bbox), so the name+type
      // combination carries enough notability signal on its own.
      if (facilityType === 'desalination' && hasName(el.tags)) {
        // skip not_notable check; proceed to score floor
      } else if (!passesCompound) {
        if (rejections) rejections.not_notable++;
        if (byTypeBucket) byTypeBucket.not_notable++;
        return null;
      }
      ```
      Document in CALIBRATION.md.

    Branch 2d — **dams below band** (< 300). Investigate byCountry. If priority countries
    are thin, the D-05 hasName requirement may be over-filtering. Options:
    - (i) Reintroduce a narrow exemption for dams with capacity data but no name in
      priority countries. Requires adding a second hasCapacityData clause to the D-05 gate.
      Write this as a D-07 exception and document.
    - (ii) Accept the count (user said "err high not low, but willing to go below 400 for
      those entities" — ~250 is still within spirit of the target).

    Prefer (ii). Do not relax D-05; the user's directive "officially recognized with a name,
    otherwise just drop it" is firm.
