# Phase 27.3.1: Water Facility Retry and Cleanup (INSERTED)

**Inserted:** 2026-04-18 during 27.3 UAT retry — Overpass API mirrors all unavailable, deferring verification + follow-up cleanup to a dedicated decimal phase.

**Revised:** 2026-04-18 (same day) — user tightened count targets (~400 dams / ~400 reservoirs / 10–25 desal), strengthened the "officially recognized" notability floor, expanded observability scope, and locked in the snapshot format + sites parity decisions.

<domain>
## Phase Boundary

Verify + calibrate water facility counts to the user's target band (~400 dams / ~400 reservoirs / 10–25 desal, each with a real OSM name), persist the result to a committed JSON snapshot so cold starts don't depend on Overpass availability, apply the same snapshot + telemetry treatment to sites, expand DevApiStatus observability, and clean up `overpass-water.ts` accumulated complexity. Architecture must scale to many concurrent users — Overpass never on the request path synchronously.

Scope is **tighten + consolidate + persist + observe**, not expand. New water layer features (heatmaps, predictive overlays, ML-based ranking) belong elsewhere.

</domain>

<decisions>
## Implementation Decisions

### R-01: Post-recovery verification

Once Overpass is available, hit `/api/water?refresh=true` on a fresh Redis (`water:facilities` + `water:precip` both cleared) and record per-type counts + filterStats. Confirm dams, reservoirs, desalination all have non-zero returns. This is the go/no-go gate for the rest of the phase.

### R-02: Count calibration — narrowed targets

- **D-01:** Target band for dams: **~400**, range acceptable from ~300 to ~500. Err high not low.
- **D-02:** Target band for reservoirs: **~400**, same ~300–500 acceptable range.
- **D-03:** Target band for desalination: **10–25** (up from the 4 currently admitted; previous run produced 13 which is the mid-band).
- **D-04:** Tune with filterStats visibility (per-type raw→kept, rejection bucket breakdown, score histogram), not blind threshold changes. If the first refresh lands outside the band, the rejection distribution tells you which gate to adjust.

Package A's first-run 892 reservoirs is ~2× the high end — expect `MIN_NOTABILITY_SCORE` climb from 15 back toward 20, or a stricter REV-2 compound gate (see R-03).

### R-03: "Officially recognized with a name" — strengthened notability floor

- **D-05:** Every admitted facility MUST have `hasName(tags)` true. No exceptions, no priority-country bypass. "Dam" / "Reservoir" / "Desalination Plant" generic-type labels never render.
- **D-06:** Admission formula (all types): `hasName(tags) AND (isNotable(tags) || isPriorityCountry(lat,lng) || hasCapacityData(tags))` where `hasCapacityData` = any of `height | volume | capacity | area` tags present.
- **D-07:** Dams in priority countries **lose** the unnamed-admission path. Previously `facilityType === 'dam' && !inPriority && !hasName(tags)` rejected — extend to reject unnamed dams in ALL countries.
- **D-08:** `MIN_NOTABILITY_SCORE` becomes a coarse secondary gate; the D-06 compound gate is the primary admission logic.
- **D-09:** Reservoir no_city rule from Plan 04/05 stays as-is (scoped to reservoirs only, priority-country-named exemption preserved). The stricter D-06 gate will drop most of the facilities no_city was catching, but the rule keeps reservoirs from admitting via weak proximity-only signals.
- **D-10:** Audit the full label fallback chain end-to-end, including `getWaterFacilityDisplayName` and the generic-token sentinel path. If D-05 holds, the "generic type fallback" codepath becomes unreachable — confirm or delete it.

### R-04: Committed JSON snapshot (src/data/water-facilities.json)

- **D-11:** Snapshot format: **pre-enriched `WaterFacility[]`**. Enrichment (capacity, nearestCity, linkedRiver, notabilityScore, stress) runs at snapshot generation time and is baked into the JSON. Cold start reads it verbatim — no runtime enrichment, no Nominatim calls on the request path.
- **D-12:** Refresh mechanism: **manual script**, `scripts/refresh-water-facilities.ts`. Developer runs it when they want fresh data; it hits Overpass, runs the full pipeline (classify → D-06 gate → enrichment → reverse-geocode unnamed → write JSON). Output is committed in the same commit as the script run.
- **D-13:** Runtime tier (request path): `Redis (canonical) → dev file cache (local dev only) → src/data/water-facilities.json (cold-start seed)`. The snapshot is the floor; Redis can drift fresher. Overpass is NEVER on a user request.
- **D-14:** No Vercel cron auto-refresh in 27.3.1. Refresh is intentional, reviewed via diff, and committed. Automation deferred to a future phase if needed.
- **D-15:** Snapshot follows the pattern of `src/data/ethnic-zones.json`, `src/data/rivers.json`, `src/data/aqueduct-basins.json`. Schema: `{ generatedAt: ISO8601, facilities: WaterFacility[], stats: WaterFilterStats }` so filter diagnostics survive in the static payload.

### R-05: Sites get full parity (snapshot + refresh script + telemetry)

- **D-16:** Add `src/data/sites.json` mirroring the water pattern. Pre-enriched `SiteEntity[]`, generated-at timestamp.
- **D-17:** Add `scripts/refresh-sites.ts` that wraps `server/adapters/overpass.ts` `fetchSites()`. Same manual-script refresh model as water — no cron.
- **D-18:** Wire `overpass.ts` through a snapshot-first tier matching the water route: `Redis → src/data/sites.json`. `useSiteFetch` cold-start miss reads the snapshot instead of hitting Overpass.
- **D-19:** Add a Sites panel to DevApiStatus with the same treatment as Water: source (snapshot/Redis/Overpass), generated-at, per-type counts, Overpass health for the last fetch attempt.

### R-06: Clean up `overpass-water.ts`

File has grown to 744 lines through Plans 01–05 + two debug rounds. Specific cleanup:

- **D-20:** Consolidate admission logic. Multiple rejection branches in `normalizeWaterElement` (excluded_location, REV-2 not_notable, low_score, no_city, no_name) each re-check `inPriority`, `hasName`, `isNotable`. Extract a single `computeAdmissionDecision(tags, coords, type) → { verdict: 'admit' | 'reject', bucket: RejectionBucket, score: number }` helper so the control flow reads top-to-bottom.
- **D-21:** Document or consolidate `DAM_IN_NAME_RE` — the terminal `\bdam\s*$` regex reclassifies reservoir-tagged facilities with "X Dam" as dams. Its reason-for-being is the 2026-04-18 debug round; capture that inline as a brief comment (not a retelling of the whole saga).
- **D-22:** Prune dead stats fields left over from Plan 03/04 consolidation. If `scoreHistogram` isn't driving any UI decision after the R-08 observability pass, drop it. If `enrichment` counts aren't read anywhere, drop them. Don't leave ghost counters.
- **D-23:** Fetch loop cleanup — the primary/fallback mirror retry has no exponential backoff, no circuit-breaker, and throws on all-mirrors-fail. Good enough for 27.3.1 (R-04 takes Overpass off the request path entirely), but add a single comment explaining the "fail loud, serve snapshot" contract.
- **D-24:** Delete ghost code; do not just rearrange. Keep commits atomic per concern — one commit per helper extraction, one commit per dead-code deletion.

### R-07: Multi-user resilience

- **D-25:** Redis is the production source of truth. Static snapshot seeds the first miss. Overpass is never on a user request path synchronously.
- **D-26:** Per-IP rate limit stays (already 10/60s in `middleware/rateLimit.ts`).
- **D-27:** No circuit breaker added in 27.3.1 — unnecessary once R-04 lands (static snapshot handles Overpass outages gracefully).

### R-08: Observability expansion (NEW)

Current DevApiStatus Water Filters panel shows raw/filtered counts, 6-bucket rejections, score histogram, enrichment coverage. User explicitly asked for improved API observability for water. Add:

- **D-28:** **Per-country admission counts.** Break `filteredCounts` by country (nearest centroid). Surfaces imbalances like "Turkey: 134, Iran: 47" instantly. Implementation: extend `WaterFilterStats` with `byCountry: Record<string, Record<WaterFacilityType, number>>`. Render as a compact table in DevApiStatus (facility type columns × country rows).
- **D-29:** **Overpass health per fetch.** Track: which mirror succeeded, HTTP status per attempt, query duration ms, total attempts before success. Extend stats with `overpass: { mirror: string, status: number, durationMs: number, attempts: number }[]` — one entry per facility-type query. Make outages visible at a glance instead of a log dive.
- **D-30:** **Snapshot provenance.** Response carries `source: 'snapshot' | 'redis' | 'overpass'` and `generatedAt: ISO8601`. DevApiStatus renders it as a small header row above the filter stats. Essential once R-04 lands.
- **D-31:** **Per-type rejection breakdown.** Current "nocity=X not_notable=Y low_score=Z" is summed across all facility types. Split by type: `reservoirs: no_city=250 low_score=120 not_notable=102 | dams: no_name=18 low_score=7 | desalination: low_score=3`. Surfaces which gate each type hits hardest.
- **D-32:** Do NOT add a sites observability panel yet in R-05 — the telemetry parity specified in D-19 reuses these four additions, so the extension points land first here, then the sites panel consumes them.

### Claude's Discretion

- Exact DevApiStatus layout/typography for the expanded observability panel
- How `computeAdmissionDecision` (D-20) is implemented (explicit state machine vs guarded early-returns) as long as it subsumes the existing scattered checks
- Whether snapshot generation uses the live server route or a standalone pipeline
- Whether the sites snapshot includes basin stress or leaves that runtime

Dev file cache shadows the snapshot for local dev speed; snapshot is the production cold-start floor.

## Specific Ideas

- "~400 dams and ~400 reservoirs, each one officially recognized with a name, otherwise just drop it." — User directive. No generic-type labels on the map ever. R-03 implements this literally.
