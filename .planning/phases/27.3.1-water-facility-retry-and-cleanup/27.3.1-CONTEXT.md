# Phase 27.3.1: Water Facility Retry and Cleanup (INSERTED)

**Inserted:** 2026-04-18 during 27.3 UAT retry — Overpass API mirrors all unavailable, deferring verification + follow-up cleanup to a dedicated decimal phase.

## Origin

Phase 27.3 shipped Plans 01–05 (filter architecture, dev diagnostics, no_city scoping, label sanitization). During the post-merge UAT, Debug round 3 (`reservoirs-missing-after-05`) applied **Package A** to `server/adapters/overpass-water.ts`:

- `PRIORITY_COUNTRIES` expanded from 7 → 14 (added Turkey, Egypt, Saudi Arabia, UAE, Kuwait, Qatar, Yemen)
- `MIN_NOTABILITY_SCORE`: 25 → 15
- REV-2 gate: `isNotable(tags) || (inPriority && hasName(tags))` → `isNotable(tags) || hasName(tags)`
- Added missing `User-Agent` header — overpass-api.de was returning HTTP 406 silently without one

Partial verification showed 892 reservoirs passing filters on the one successful Overpass response before all three mirrors (`overpass-api.de`, `overpass.private.coffee`, `overpass.kumi.systems`) went fully unavailable at ~15:15 PT. Full dam/desal verification blocked.

## Goal

Verify + tighten Package A counts to the user's target band, persist the result to a committed JSON snapshot to decouple cold-starts from Overpass availability, and clean up `overpass-water.ts` accumulated complexity. Architecture must scale to many concurrent users without hammering Overpass.

## Requirements

### R-01: Post-recovery verification

Once Overpass is available, hit `/api/water?refresh=true` on a fresh Redis (`water:facilities`, `water:precip` both cleared) and record per-type counts + filterStats. Confirm dams, reservoirs, desalination all have non-zero returns.

### R-02: Count calibration

**Target:** ~100–500 dams AND ~100–500 reservoirs (range not a point — any figure in this band is acceptable; err high not low). Desalination: rediscover the original 13 coverage (currently 4).

Package A first-run suggested 892 reservoirs, which is ~2× the high end of target. Likely need to restore `MIN_NOTABILITY_SCORE` to ~20 or add a named-and-scored compound gate. Tune with filterStats visibility, not blind threshold changes.

### R-03: Every facility is significant and named

No generic "Dam near Unknown", "Reservoir near Unknown", etc. ever appear on the map, in tooltips, in the detail panel, in counters, or in search results. If a facility lacks an OSM `name` AND reverse geocoding cannot resolve a city, **drop it** rather than emit a generic label. Audit the full label fallback chain end-to-end, including `getWaterFacilityDisplayName` and any generic-token sentinel paths from Plan 05.

### R-04: Committed JSON snapshot of water facilities

Follow the pattern of `src/data/ethnic-zones.json`, `src/data/rivers.json`, `src/data/aqueduct-basins.json`. Produce `src/data/water-facilities.json` (or a server-side equivalent that's checked into the repo and versioned) as the **cold-start seed**. Runtime tier stays: Redis (canonical, serverless-friendly) → dev file cache (local dev only) → static JSON (cold start + build-time). Refresh the static snapshot on demand via a script (e.g. `scripts/refresh-water-facilities.ts`), not every Vercel deploy.

### R-05: Sites get the same treatment if not already

Audit `server/adapters/overpass.ts` and `siteStore`. If sites aren't already backed by a committed snapshot + Redis, add it. Same tier: snapshot seed → Redis canonical → one-time fetch on mount.

### R-06: Clean up `overpass-water.ts`

The file has grown through Plans 01–05 + two debug rounds. Specific cleanup candidates:

- Multiple rejection branches in `normalizeWaterElement` with overlapping predicates (e.g. the no_city / REV-2 / low_score / excluded_location paths each re-check inPriority)
- Scattered `isNotable` / `hasName` / `isPriorityCountry` calls — consider a small `computeAdmissionDecision(tags, coords, type) -> { verdict, bucket }` helper
- `DAM_IN_NAME_RE` override logic — now lives alongside standard classification; document or consolidate
- Dead or unused stats fields after Plan 03/04 consolidation
- Multi-mirror fetch loop retry semantics (no exponential backoff, all-mirrors-fail throws)

Delete ghost code. Do not just rearrange. Keep commits atomic per concern.

### R-07: Multi-user resilience

Assumption: the app will serve many concurrent users. Redis is the source of truth in production; static snapshot seeds the first miss; Overpass is never on a user's request path synchronously. Consider: background cron refresh (Vercel cron hits `/api/water?refresh=true` with the cron-identifying UA), circuit breaker on repeated Overpass failures, per-IP rate limit preserved (already at 10/60s in `middleware/rateLimit.ts`).
