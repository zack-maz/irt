## Notes for Plan 08 (DevApiStatus Sites panel — the UI layer)

Plan 08 consumes `siteStore.filterStats` to render a `SitesFiltersSection` in DevApiStatus. Same treatment as the existing `WaterFiltersSection`. Field-by-field rough shape the panel should render:

| Field                        | Location                                | Render pattern suggestion (mirrors water panel)                                                                                                                                                                       |
| ---------------------------- | --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `source`                     | `filterStats.source`                    | Provenance header row: `Source: snapshot · Generated: <relativeTime(generatedAt)>` — reuse the `relativeTime` helper already in DevApiStatus                                                                          |
| `generatedAt`                | `filterStats.generatedAt`               | ISO → relative time via existing helper                                                                                                                                                                               |
| `rawCount` → `filteredCount` | `filterStats.{rawCount, filteredCount}` | `"{rawCount} raw → {filteredCount} kept ({filteredCount/rawCount*100}%)"` — matches water panel line 2                                                                                                                |
| `byType`                     | `filterStats.byType`                    | Per-type row: `nuclear=45 naval=60 oil=99 airbase=284 port=232` — sorted by count desc for scannability                                                                                                               |
| `byCountry`                  | `filterStats.byCountry`                 | Country table keyed by nearestCountry → per-SiteType counts. Top-12 cap (matches water panel's byCountry rendering — T-27.3.1.03-04 DoS guard). Use same table component.                                             |
| `rejections`                 | `filterStats.rejections`                | Rejection row: `excluded_turkey=156 no_coords=0 no_type=0 duplicate=0` — 4 buckets only (sites has narrower rejection surface than water; do NOT add placeholder buckets like `low_score`/`no_city` that don't apply) |
| `overpass`                   | `filterStats.overpass`                  | One row per attempt: `sites · primary · status=200 · 17524ms · attempts=1 · OK`. Red on FAIL. Same format as water's Overpass Health block.                                                                           |

**Do NOT mirror water's byTypeRejections for sites** — sites has a single query that touches all 5 types, so a `byTypeRejections` split would require restructuring the fetch loop. The current sites `rejections` is already narrower and type-agnostic (an excluded_turkey sample might have been any type). Plan 08 can note this asymmetry in a comment but should not invent per-type buckets.

## R-05 Observability Asymmetry (Sites vs Water)

Sites' 4-bucket rejection register is NARROWER than water's 6-bucket because the sites adapter is simpler (single Overpass query, no compound admission gate, no scoring, no nearestCity requirement). Specifically, sites does NOT have:

- `no_name` — extractLabel has a generic-type fallback (SITE_TYPE_LABELS), so all sites get a non-empty label. Unlike water's hasName(tags) mandatory gate.
- `not_notable` / `low_score` — no notability scoring on sites. OSM presence + correct tag = admit.
- `no_city` — no nearestCity requirement on sites (water uses it for reservoir labeling).

This asymmetry is **by design** — sites are higher-value-per-record than water and the OSM tag schema for infrastructure is more canonical. Don't add synthetic buckets.
