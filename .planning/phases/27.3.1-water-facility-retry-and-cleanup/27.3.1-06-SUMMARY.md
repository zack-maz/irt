### computeAdmissionDecision helper (R-06 D-20)

```typescript
export type AdmissionVerdict =
  | { verdict: 'admit' }
  | { verdict: 'reject'; bucket: keyof WaterFilterStats['rejections'] };

export function computeAdmissionDecision(
  tags: Record<string, string>,
  lat: number,
  lng: number,
  facilityType: WaterFacilityType,
  inPriority: boolean,
  score: number,
  nearestCity: ReturnType<typeof findNearestCity>,
): AdmissionVerdict {
  // 1. excluded_location
  // 2. no_name (R-03 D-05/D-07)
  // 3. not_notable (2-of-3 compound, exempted for desalination)
  // 4. low_score (R-03 D-08 secondary floor)
  // 5. no_city (reservoir-scoped, Plan 27.3-05)
  // 6. admit
}
```

Implemented as **guarded early-returns** (not a state machine). Matches the short-circuit semantics of the pre-refactor scattered branches exactly. Reads top-to-bottom. Pure — no side effects, no logging, returns structured verdict for caller to tally.

### Fetch-loop fail-loud contract documented (R-06 D-23)

Added to `fetchWaterFacilities` JSDoc:

> **Failure contract (Phase 27.3.1 R-06 D-23) — fail loud, serve snapshot:**
> Each facility-type query tries primary then fallback Overpass mirror. If a specific type fails on both mirrors, that type contributes zero facilities but the function continues. If all three queries fail against both mirrors, this throws.
>
> No circuit breaker, no exponential backoff. Rationale: Phase 27.3.1 R-04 (committed snapshot) means a cold user request never reaches this function synchronously — the water route serves Redis → devFileCache → snapshot before trying Overpass, and only explicit refresh paths (`npm run refresh:water` or `?refresh=true` with the dev/cron gate) invoke this.

Surviving call sites benefit from failing loud — the refresh script surfaces errors to the developer; the refresh-bypass route falls through to stale Redis.

## Decisions Made

1. **Guarded early-returns over state machine.** The pre-refactor code used short-circuit returns from within `normalizeWaterElement`. A state machine would add ceremony without removing any branches — the decision IS inherently sequential, not parallel-branching.

2. **AdmissionVerdict is a discriminated union keyed off `verdict`.** `{ verdict: 'admit' } | { verdict: 'reject'; bucket: keyof WaterFilterStats['rejections'] }`. The bucket key type is derived from the stats contract via `keyof`, so adding a new rejection bucket to `WaterFilterStats['rejections']` automatically becomes valid bucket territory here — but that new bucket is NOT automatically emitted (the helper body would need an explicit branch). This is the safe default.

3. **`no_city` branch kept as defense-in-depth despite being unreachable.** Under Plan 04's 2-of-3 compound gate + hasName prerequisite + reservoir scoping, the path from "not already rejected" to "no_city fires" requires a combination of inputs (priority=true, hasCapacityData=true, hasName=true, !hasWikiRef, !isNamedInPriorityCountry) where `inPriority && hasName` always yields `isNamedInPriorityCountry=true`. So no_city is effectively dead. A comment block above the exemption tests documents this explicitly. Keeping the branch is cheaper than the risk of re-introducing a reachable no_city admission path in a future relaxation.

4. **All 10 WaterFilterStats fields preserved.** DevApiStatus renders every one. The plan's own "when in doubt, KEEP" heuristic plus the explicit DevApiStatus references nail the decision.

5. **No snapshot regeneration needed.** `src/data/water-facilities.json` contains a `stats` sub-object with `scoreHistogram` and `enrichment` — but since those fields were NOT pruned, the snapshot schema is unchanged. No regeneration required.
