# Phase 27.3.1 Plan 03: R-08 Observability Expansion — COMPLETE

## One-liner

Extended the `WaterFilterStats` diagnostic contract end-to-end across TS interface

- Zod schema + adapter population + route provenance + client store + DevApiStatus
  UI with four new observability fields (byCountry, overpass telemetry, source +
  generatedAt provenance, byTypeRejections), wiring the snapshot tier value now so
  Plan 05 can populate it without reshaping the contract.

## Extended interface (field-by-field)

```typescript
export interface WaterFilterStats {
  // Pre-R-08 fields (unchanged):
  rawCounts: Record<string, number>;
  filteredCounts: Record<string, number>;
  rejections: {
    /* six summed buckets */
  };
  enrichment: { withCapacity: number; withCity: number; withRiver: number };
  scoreHistogram: { bucket: string; count: number }[];

  // Phase 27.3.1 R-08 D-31 — per-facility-type rejection breakdown.
  // Sum over byTypeRejections.* matches `rejections` (lock-step).
  byTypeRejections: Record<
    string,
    {
      excluded_location: number;
      not_notable: number;
      no_name: number;
      duplicate: number;
      low_score: number;
      no_city: number;
    }
  >;

  // Phase 27.3.1 R-08 D-28 — admitted facilities keyed by nearest-centroid
  // country → facilityType → count. Two-level map.
  byCountry: Record<string, Record<string, number>>;

  // Phase 27.3.1 R-08 D-29 — Overpass fetch telemetry, one entry per URL
  // attempt. Empty array when data came from snapshot/redis.
  overpass: OverpassFetchRecord[];

  // Phase 27.3.1 R-08 D-30 — provenance tag.
  source: 'snapshot' | 'redis' | 'overpass';

  // Phase 27.3.1 R-08 D-30 — ISO-8601 timestamp of when the data was produced.
  generatedAt: string;
}

export interface OverpassFetchRecord {
  facilityType: string; // 'dams' | 'reservoirs' | 'desalination'
  mirror: string; // 'primary' | 'fallback' (label, NOT raw URL)
  status: number; // HTTP status; 0 when network error before response
  durationMs: number;
  attempts: number; // 1 on first-try success, 2 when fallback hit
  ok: boolean; // final outcome
}
```

## Where each field is populated

| Field                                | Populated at                                                                                                                                                                 | Notes                                                              |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| `rawCounts` / `filteredCounts`       | `fetchFacilityType` (per-query, after JSON parse)                                                                                                                            | Unchanged from pre-R-08.                                           |
| `rejections.*`                       | `normalizeWaterElement` (six rejection sites)                                                                                                                                | Unchanged from pre-R-08.                                           |
| `byTypeRejections.<facilityType>.*`  | `normalizeWaterElement` via new `byTypeBucket` parameter; bucket initialized in `fetchFacilityType` BEFORE the network call so the key exists even when zero rejections fire | Six dual-write sites mirror `rejections.*` increments.             |
| `byCountry.<country>.<facilityType>` | `fetchWaterFacilities` after dedup, fused with enrichment-coverage loop                                                                                                      | Uses new `nearestCountryName(lat, lng)` exported helper.           |
| `overpass[]`                         | `fetchFacilityType` per URL attempt (push on both success and failure)                                                                                                       | Records mirror label, HTTP status, duration, attempts, ok outcome. |
| `source`                             | `fetchWaterFacilities` initial = `'overpass'`; route layer overwrites to `'redis'` for cached/error-with-cache/dev-file-cache paths                                          | Plan 05 will set `'snapshot'` when the JSON tier lands.            |
| `generatedAt`                        | `fetchWaterFacilities` initial stamp + refresh at completion; route layer derives from `cached.lastFresh` for cached paths                                                   | ISO 8601.                                                          |

## Plan 05 / shape-compat notes

`source: 'snapshot' | 'redis' | 'overpass'` is wired in the type union NOW so
Plan 05 (R-04 snapshot tier) can simply set `source = 'snapshot'` when serving
from the committed JSON tier without reshaping `WaterFilterStats`. Plan 05
changes that this plan deliberately does NOT block on:

1. The dev-file-cache fallback branch in `server/routes/water.ts` is currently
   labeled `source='redis'` with a TODO comment. When Plan 05 lands, that
   branch should switch to `source='snapshot'` because dev-file-cache is
   semantically closer to the snapshot tier than to Redis.
2. The error-without-cache branch attaches an empty `overpass[]` because
   `fetchWaterFacilities` throws before stats are returned. Once Plan 05
   adds the snapshot tier as a cold-start fallback, this branch will become
   much rarer and the gap stops mattering. If Plan 05 wants to expose the
   partial fetch telemetry on the error path, it will need to widen
   `fetchWaterFacilities` to attach `stats` to the thrown error.
3. The `WaterDataSource` type alias is exported from
   `server/adapters/overpass-water.ts` — Plan 05 can import it for its
   snapshot loader.

## Plan 08 / sites parity notes

Per CONTEXT.md D-32, sites observability (R-05 / D-19) reuses the four
extension points landed here. The patterns Plan 08 should mirror:

1. `OverpassFetchRecord` → can be reused (or re-declared with the same
   shape) for sites — same Overpass-API contract.
2. `nearestCountryName` is exported as a shared helper — sites can reuse it
   for site-side `byCountry`.
3. `buildEmptyFilterStats` helper pattern in `server/routes/water.ts` is
   the model for `server/routes/sites.ts` — every response path attaches
   provenance, only the tier transition stub differs.
4. The DevApiStatus.WaterFiltersSection's null-safe placeholder path
   (renders "loading filter stats…" when filterStats is absent) is
   preserved and is the model for the SitesFiltersSection's defensive
   fallback. It IS still reachable (pre-fetch) so Plan 08 must keep this
   pattern.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Test fixture coords for Turkey didn't actually map to Turkey's centroid**

- **Found during:** Task 2 GREEN phase verification.
- **Issue:** Initial test used SE Turkey coords (37.9, 40.2) for `turkeyReservoir`, expecting `byCountry.Turkey` to be set. But nearest-centroid attribution (Iraq=611km, Turkey=452km, Syria=378km) actually picks Syria. Rewrote with East-Turkey coords (39.5, 40.0) where Turkey IS nearest centroid AND distFromSE (179km) survives the western-Turkey exclusion (600km cap).
- **Fix:** Updated `turkeyReservoir` fixture coords + comment explaining the attribution honesty.
- **Files modified:** `server/__tests__/adapters/overpass-water.test.ts`.
- **Commit:** `24a24ee` (bundled with Task 2 GREEN since the fixture is integral to the test).

**2. [Rule 2 — Critical] Zod schema needed `.strict()` to make missing R-08 fields a validation failure**

- **Found during:** Task 1 RED phase test inspection — `rejects unknown source value` failed RED because Zod was silently dropping the unknown enum value rather than failing.
- **Fix:** Added `.strict()` to the inner schema object so unknown fields fail. Outer `.optional()` preserved (truth-21 — cached payloads with NO filterStats key still validate).
- **Decision rationale:** without `.strict()`, a future bug that drops one of the new fields from the response would silently degrade observability — exactly the failure mode R-08 is fixing. `.strict()` makes it a loud failure at the schema layer.
- **Files modified:** `server/schemas/cacheResponse.ts`.
- **Commit:** `313bc26` (Task 1 GREEN).

**3. [Rule 3 — Scope] Refactored `isPriorityCountry` and `isExcludedLocation` to share `nearestCountryName`**

- **Found during:** Task 2 implementation — needed `nearestCountryName` as an exported helper for byCountry tallying anyway, and the same nearest-centroid loop existed in three places (isPriorityCountry, isExcludedLocation, planned byCountry tally).
- **Fix:** Extracted `nearestCountryName` once and rewrote `isPriorityCountry` (one-liner now) and `isExcludedLocation` (uses nearestCountryName instead of duplicated loop).
- **Files modified:** `server/adapters/overpass-water.ts`.
- **Commit:** `24a24ee` (Task 2 GREEN).
- **Risk:** none — the extracted function has identical behavior to the inline loops. All 99 existing adapter tests still pass.

### Out-of-scope (logged, not fixed)

- Pre-existing client-side test failures (34 in `filters.test.ts` / `entityLayers.test.ts` / `devApiStatus.test.tsx`) — same count as Plan 02 baseline, no Plan 03 regression. Already tracked in `deferred-items.md`.

### NOT a deviation — interface location decision

The plan frontmatter's `artifacts` list says `server/types.ts` should contain
the extended `WaterFilterStats`. The interface was kept in
`server/adapters/overpass-water.ts` instead (where it has lived since Phase 27.3).
This is a preserved Phase 27.3 convention (truth-12 rationale) — interfaces
owned by the adapter that produces them, with client stores re-declaring for
tier independence. Moving it to `server/types.ts` would require the adapter
to import from types.ts (creating a backward dep that Phase 27.3 deliberately
broke). The TypeScript module surface is identical; only the file location
differs from the plan's artifact path.
