## Summary

This phase covers two independent improvement tracks: (1) event data quality via tiered news source filtering, richer LLM enrichment inputs, and three UX fixes (precision ring opacity, zoom icon disappearance, date slider styling), and (2) water facility coverage expansion via Overpass query tuning with reverse geocoding for unnamed facilities, filter parity, and icon sizing.

The codebase already has strong foundations for both tracks. The news pipeline (`gdelt-doc.ts` -> `newsFilter.ts` -> `relevanceScorer.ts`) already has a 4-tier `SOURCE_RELIABILITY` map in the relevance scorer. The LLM extractor has a clear `BATCH_SIZE` constant and `buildBatchUserPrompt` function where richer input can be injected. The water Overpass adapter has explicit `["name"]` tags on 3 of 4 query types that can be removed. Nominatim already has both `forwardGeocode` and `reverseGeocode` functions wired up.

**Primary recommendation:** Execute as two parallel workstreams (events quality + water data), with the three UX fixes as a third small workstream. All changes are localized to well-defined files with clear insertion points.

### Pattern 5: Zoom Icon Disappearance Root Cause

**What:** Event icons disappear at high zoom. The `sizeMaxPixels` for event types ranges from 120 to 160 pixels.

**Root cause analysis:** When zooming in with `sizeUnits: 'meters'`, icons grow proportionally to the map scale. At high zoom, the icon's meter-based size converts to many pixels, which gets capped by `sizeMaxPixels`. However, `sizeMaxPixels` should clamp the icon to that pixel size, not hide it. The icons should still be visible at maxPixels.

**More likely cause:** The `billboard: false` setting combined with terrain/perspective can cause icons to be occluded or face away from the camera at high zoom. Another possibility is that the icon atlas is canvas-based and may have resolution issues at large pixel sizes.

**Diagnostic steps:**

1. Check if `billboard: false` is the issue (test with `billboard: true`)
2. Check if the icon atlas resolution is too low for the maxPixels scale
3. Check if there's a deck.gl minimum size threshold that makes very-high-zoom meter-based icons collapse
4. Verify the issue doesn't affect flights/ships (which use the same pattern)

This needs hands-on diagnosis. The planner should include an explicit diagnostic task before the fix task. [ASSUMED]

## Common Pitfalls

### Pitfall 1: GDELT DOC Domain Field Not Propagated

**What goes wrong:** The `domain` field from GDELT DOC API responses is parsed but never mapped to `NewsArticle` -- it's silently dropped in `gdelt-doc.ts`.
**Why it happens:** The `NewsArticle` type doesn't have a `domain` field, and the mapping function only extracts explicit fields.
**How to avoid:** Either add `domain` to `NewsArticle` or extract domain from URL using `new URL(article.url).hostname` at classification time.
**Warning signs:** Tier classification for GDELT articles returns "unknown" for everything because domain is missing.

### Pitfall 2: Nominatim Rate Limiting for Unnamed Water Facilities

**What goes wrong:** Removing the `["name"]` filter from Overpass queries could yield hundreds of unnamed facilities. Each needs a reverse geocode call. At 1 req/s Nominatim rate limit, 300 unnamed facilities = 5 minutes of sequential geocoding.
**Why it happens:** Nominatim's usage policy requires max 1 request per second.
**How to avoid:** (1) Use Redis caching for reverse geocode results (already done for forward geocode -- `geocode:${lat},${lon}` key with 30-day TTL). (2) Pre-batch the geocoding during the water route handler, not on every request. (3) The water endpoint already has 24h Redis TTL, so geocoding only runs once per day at most.
**Warning signs:** Water route takes >60s to respond on cold cache.

### Pitfall 3: Overpass Query Explosion Without Name Filter

**What goes wrong:** Removing `["name"]` from dam and treatment plant queries could return thousands of tiny unnamed dams, farm ponds, etc. that are noise.
**Why it happens:** OSM tags many minor features (irrigation dams, small weirs) with `waterway=dam` but no `name` tag.
**How to avoid:** Keep the existing `isPriorityCountry()` and `isNotable()` filtering in `normalizeWaterElement()`. For priority countries, unnamed facilities still pass through (desired). For non-priority countries, the notability check (wikidata/wikipedia tag) already filters out minor features. Consider adding a minimum `way` size threshold for unnamed dams.
**Warning signs:** Water endpoint returns 5000+ facilities, most being tiny unnamed features.

### Pitfall 4: Precision Ring Layer Performance with Per-Entity Alpha

**What goes wrong:** Switching from a static `getFillColor` to a per-entity accessor that checks `selectedEntityId` on every render cycle can cause performance issues if there are many events.
**Why it happens:** deck.gl recalculates accessors for every datum when updateTriggers change.
**How to avoid:** Use `updateTriggers: { getFillColor: [selectedEntityId] }` so the accessor only re-runs when selection changes. With typical event counts (50-200), this is not a real concern.
**Warning signs:** Lag when selecting/deselecting events with many precision rings visible.

### Pitfall 5: LLM Batch Size Reduction Increases API Calls

**What goes wrong:** Cutting batch size from 8 to 4 doubles the number of LLM API calls for the same number of event groups.
**Why it happens:** Each API call has overhead (network latency, rate limits on free tier).
**How to avoid:** The locked decision acknowledges this: "With fewer junk events post-filtering, total API calls may decrease despite smaller batches." The tier-based filtering should reduce event group count enough to offset the smaller batches. Monitor total calls via the Phase 27.1 observability endpoint (`/api/events/llm-status`).
**Warning signs:** LLM pipeline takes significantly longer or hits rate limits more frequently.

### Severity Score with Source Tier Weighting

```typescript
// Modified severity.ts
// Source: codebase analysis of existing computeSeverityScore

// New: source tier multiplier
const SOURCE_TIER_MULTIPLIER: Record<number, number> = {
  1: 1.5, // Gold sources boost
  2: 1.0, // Silver sources baseline
  3: 0.7, // Bronze sources demote
};

export function computeSeverityScore(event: ConflictEventEntity): number {
  const typeWeight = TYPE_WEIGHTS[event.type] ?? 3;
  const mentions = event.data.numMentions ?? 1;
  const sources = event.data.numSources ?? 1;

  const ageMs = Math.max(0, Date.now() - event.timestamp);
  const ageHours = ageMs / (1000 * 60 * 60);
  const recencyDecay = 1 / (1 + ageHours / 24);

  // Source tier multiplier (requires tier data on entity)
  const tierMultiplier = SOURCE_TIER_MULTIPLIER[event.data.sourceTier ?? 2] ?? 1.0;

  return (
    typeWeight * Math.log2(1 + mentions) * Math.log2(1 + sources) * recencyDecay * tierMultiplier
  );
}
```

### 7. Entity Layer Icon Disappearance — sizeMaxPixels Analysis

All event entity layers use `sizeUnits: 'meters'` with these maxPixels caps:

- airstrike: 160px, on_ground: 120px, explosion: 160px, targeted: 140px, other: 120px
- flights: 100px, ships: 100px, sites: 50px

At very high zoom (e.g., zoom 18+), meter-based icons become many thousands of pixels, clamped to maxPixels. The icons should still render at maxPixels size -- they should NOT disappear.

**Possible causes (ranked by likelihood):**

1. **`billboard: false` + perspective view**: At high zoom with terrain, non-billboard icons orient to the globe surface and may become edge-on or clip through terrain. This is the most likely cause.
2. **Icon atlas resolution**: The canvas-based icon atlas may look blurry/invisible at very large pixel sizes if the atlas texture is too small.
3. **Data filtering**: Some zoom-dependent filter removes events from the data array.

**Recommended diagnostic:** Toggle `billboard: true` on one event layer and test at high zoom. If icons reappear, the fix is either `billboard: true` for event layers (they don't need heading rotation like flights) or increasing maxPixels. [ASSUMED -- needs hands-on verification]

`src/lib/severity.ts` has two functions:

- `computeSeverityScore(event)` -- used in notification center (has recency decay)
- `classifySeverity(event)` -- used for filter severity toggles (no recency decay)

Both use `TYPE_WEIGHTS[event.type]` as the base multiplier. Source tier weighting can be added as an additional multiplier to both functions.

**Challenge:** The `ConflictEventEntity` data structure currently has no source tier information. The tier needs to be computed server-side during event processing and stored on the entity's `data` object (e.g., `data.sourceTier?: 1 | 2 | 3`). The events route would set this based on the highest-tier source URL associated with the event group.

Alternatively, for the minimum viable implementation specified in CONTEXT.md: "Events with only 1 Tier 3 source get demoted. Events with 2+ Tier 1/2 sources get boosted" -- this could be a post-processing step in the events route that adjusts `numMentions` or `numSources` values, avoiding schema changes. However, a cleaner approach is adding the tier field. [VERIFIED: codebase inspection]

## Open Questions

1. **How many unnamed water facilities will Overpass return?**
   - What we know: Current named-only queries return ~100 facilities. Unnamed dams/treatment plants are common in OSM for the Middle East.
   - What's unclear: Could be 200 or 2000. The priority-country + notability filters will help but exact count unknown.
   - Recommendation: Run the modified Overpass query locally before deploying to gauge volume. Add a hard cap (e.g., 500) if needed.

2. **How to connect news article titles to GDELT event groups for LLM enrichment?**
   - What we know: Event groups have `sourceUrls[]` from GDELT row-level source fields. News cache has articles with URLs and titles.
   - What's unclear: Whether GDELT event source URLs overlap significantly with GDELT DOC article URLs.
   - Recommendation: Match by URL first. If overlap is low, fall back to matching by geographic proximity + time window between events and articles.

3. **Should sourceTier be stored on ConflictEventEntity.data or computed client-side?**
   - What we know: Server has access to source URLs; client does not.
   - What's unclear: Whether adding a field to the entity data structure is preferable to a lookup function.
   - Recommendation: Store on entity data server-side (lightweight, one-time computation). This is cleaner than client-side URL parsing.
