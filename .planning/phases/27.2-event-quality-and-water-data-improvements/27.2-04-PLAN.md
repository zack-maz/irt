**Overpass Query Changes (D-06):**

   - dams: `'nwr["waterway"="dam"]["name"]'` -> `'nwr["waterway"="dam"]'` (remove `["name"]`)
   - reservoirs: Remove `["name"]` from both the `way` and `relation` parts:

   - treatment_plants: `'nwr["man_made"="water_works"]["name"]'` -> `'nwr["man_made"="water_works"]'` (remove `["name"]`)
   - desalination: UNCHANGED (already has no name filter)

2. The existing `normalizeWaterElement()` already handles unnamed facilities through its `extractLabel()` function (falls back to generic type label like "Dam"). But per D-06, unnamed facilities should get "Dam near [city]" labels via reverse geocoding.

3. Since reverse geocoding is async and rate-limited (1 req/s), it should NOT be done inside normalizeWaterElement (synchronous). Instead, add a post-processing step.

**Reverse Geocode Labeling (D-06):**

4. Modify `server/routes/water.ts` (or create a helper in overpass-water.ts):
   - After `fetchWaterFacilities()` returns the array, identify facilities with generic labels (no OSM name tag). These are facilities where `label` is just the facility type name (e.g., "Dam", "Reservoir", "Water Treatment Plant").
   - Create a helper function `labelUnnamedFacilities(facilities: WaterFacility[]): Promise<WaterFacility[]>`:
     - Import `reverseGeocode` from `../adapters/nominatim.js`
     - Define `FACILITY_TYPE_LABELS` map: `{ dam: 'Dam', reservoir: 'Reservoir', desalination: 'Desalination Plant', treatment_plant: 'Treatment Plant' }`
     - Identify unnamed: `f.label === FACILITY_TYPE_LABELS[f.facilityType] || !f.label.includes(' ')`
       - Actually, better check: if the label has no space or equals the type label exactly, it's unnamed.
       - More robust: Check if the label was set from OSM name tag by checking if it differs from the generic type label. The `extractLabel` function in overpass-water.ts uses `tags.name`, `tags['name:en']`, or fallback. If there's no name tag, the label is just the type name. The simplest detection: check `f.label` against known type labels.
     - For each unnamed facility (up to 500 to prevent runaway):
       - Call `reverseGeocode(f.lat, f.lng)` (already cached with 30-day TTL at 2-decimal quantized coords)
       - Set `f.label = "${typeLabel} near ${geo.city ?? geo.country ?? 'Unknown'}"`
       - Respect 1 req/s rate limit: sequential processing with 1s delay between uncached calls
       - The Redis cache means most calls after the first run will be instant (same quantized coords)
     - Return the modified array
   - Call this AFTER fetchWaterFacilities and BEFORE caching the result in Redis. This way, the 24h cache stores the labeled facilities and reverse geocoding only runs on cold cache (once per day max).

5. Add a volume safeguard: If removing name filters produces more than 800 facilities, log a warning but still serve them. The existing geographic filtering (isPriorityCountry, isExcludedLocation, isNotable) should keep volumes reasonable.
