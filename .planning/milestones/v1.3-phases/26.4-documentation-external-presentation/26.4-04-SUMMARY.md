## Decisions Made

1. **Agentic capture over manual recording.** The plan's original Task 1 instructed the user to manually screen-record with Kap or QuickTime and convert with gifski. Mid-execution I rejected this and built an agentic alternative using Playwright (already a devDep). Manual recording is not repeatable, drifts with every UI change, and requires the user to re-do the work every time. An agentic script is permanent portfolio tooling. The regeneration time is ~45 seconds.
2. **Playwright recordVideo does not work for WebGL in headless + software-GL mode.** This was the first iteration's dead end — a 55-second webm was produced but every frame was black. Playwright's `recordVideo` context option captures the browser compositor output, and in headless + SwiftShader mode the compositor receives zeroed frames for WebGL canvases. `page.screenshot()` reads the canvas backbuffer synchronously and works correctly. Documented in the script header and in the deviations section below.
3. **Skip ffmpeg in the gifski pipeline.** The second iteration tried `ffmpeg -f yuv4mpegpipe | gifski` to convert PNG frames but hit pixel-format errors (`ffmpeg` needed `-pix_fmt yuv420p` for rgb24 → yuv4mpegpipe). The third iteration skipped ffmpeg entirely — gifski accepts PNG stdin and handles lanczos scaling via `--width`. This produced a clean 1354 KB GIF on the first run.
4. **MapDevExposer dev-only component.** Rather than making the map exposure conditional at the hook level, a tiny React component gated by `import.meta.env.DEV` sets `window.__map = current` in a `useEffect`. Vite tree-shakes this entirely out of production builds. Zero bundle impact.
5. **rateLimiters.public wired BEFORE per-endpoint limiters.** Wiring the cheap baseline limiter first means scraper bursts trip it on the 7th request and never consume per-endpoint rate budgets. Per-endpoint limits remain available for legitimate high-volume users (e.g., a tab that's been open for hours).
6. **Live demo URL left as `_TBD_` placeholder.** A hardcoded Vercel production URL was replaced with a placeholder at the user's request (commit `bd453cf`). The user will substitute the real URL at publication time rather than committing it mid-plan.
7. **Screenshot capture order reshuffled.** Initial run captured detail-panel.png before search-modal.png, but leftover threat-cluster selection state from detail-panel bled into the search modal. Reordered so search-modal runs first from a clean state.
8. **Task order reversed.** Plan tasks 3 and 4 (rate limits + README rewrite) ran BEFORE Task 1 (hero GIF capture) because they were GIF-independent and unblocked the longest-lead-time artifact. The README write used placeholder links to `docs/hero.gif` while the GIF itself was still being iterated on.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Playwright recordVideo produces black frames for WebGL in headless software-GL mode**

- **Found during:** Task 1 (hero GIF capture, iteration 1)
- **Issue:** The plan assumed manual screen recording. The agentic alternative using `page.context().recordVideo = { dir, size }` successfully captured a 55-second webm, but every frame was black — Playwright's video recording reads from the browser compositor, and in headless + SwiftShader mode the compositor receives zeroed frames for WebGL canvases.
- **Fix:** Switched to a frame-sequence approach — `page.screenshot()` at 150 ms intervals during the flyTo animation (page.screenshot reads the canvas backbuffer synchronously and works correctly for WebGL), stitched into a GIF by gifski.
- **Files modified:** `scripts/capture-hero.ts`
- **Verification:** Generated a 55-frame PNG sequence, stitched it into a GIF, manually opened the GIF — layers visible, zoom legible, loop smooth.
- **Committed in:** `8f3ce82` (Task 1b commit)

**2. [Rule 3 — Blocking] ffmpeg yuv4mpegpipe pixel format error**

- **Found during:** Task 1 (hero GIF capture, iteration 2)
- **Issue:** The first version of the frame-stitching pipeline used `ffmpeg -f yuv4mpegpipe -i - | gifski -` to pipe frames through ffmpeg for lanczos scaling before handing off to gifski. ffmpeg rejected the pipe with a pixel format error — rgb24 PNGs need explicit `-pix_fmt yuv420p` when converting through yuv4mpegpipe, and the filter_complex incantation was fighting the pipe buffer size.
- **Fix:** Skipped ffmpeg entirely. gifski accepts PNG frames directly on stdin and handles lanczos scaling via `--width 1280`. Reduced pipeline from 3 processes (Playwright → ffmpeg → gifski) to 2 (Playwright → gifski).
- **Files modified:** `scripts/capture-hero.ts`
- **Verification:** Pipeline produced a 1354 KB GIF on the first run. ffmpeg dependency is now vestigial in the prerequisite check (still enforced but unused — left as-is; trimming is a minor follow-up).
- **Committed in:** `8f3ce82` (Task 1b commit)

**3. [Rule 3 — Blocking] LayerTogglesSlot sidebar not clickable when Layers panel is closed**

- **Found during:** Task 1 (hero GIF capture)
- **Issue:** Attempting to toggle individual layer switches via `page.getByLabel('Political')` failed with "element not clickable" — the Sidebar Layers content panel has `!pointer-events-none` applied when the panel is closed.
- **Fix:** The capture script now first clicks the "Layers" icon-strip button via `getByRole('button', { name: 'Layers', exact: true })` to open the panel before interacting with individual switches.
- **Files modified:** `scripts/capture-hero.ts`
- **Verification:** Layer toggles fire correctly in subsequent runs; all 6 layers activate before the flyTo sequence.
- **Committed in:** `8f3ce82` (Task 1b commit)

**4. [Rule 3 — Blocking] Cmd+K keyboard shortcut unreliable in headless chromium**

- **Found during:** Task 1 (hero GIF capture, search-modal screenshot)
- **Issue:** `page.keyboard.press('Meta+k')` to open the search modal did not register reliably in headless chromium. The modal sometimes opened, sometimes didn't. Intermittent failures are the worst kind of bug in capture scripts — one flaky run corrupts all downstream screenshots.
- **Fix:** The script now clicks `[data-testid="topbar-search-hint"]` directly, which calls `useSearchStore.getState().openSearchModal()` via the onClick handler. Deterministic.
- **Files modified:** `scripts/capture-hero.ts`
- **Verification:** Search modal opens on every run; screenshot captures the autocomplete dropdown reliably.
- **Committed in:** `8f3ce82` (Task 1b commit)

**5. [Rule 1 — Bug] Weather layer label is "Climate" in LayerTogglesSlot, not "Weather"**

- **Found during:** Task 1 (hero GIF capture, layer toggle phase)
- **Issue:** The capture script's `ALL_LAYERS` constant listed `"Weather"` to match the plan spec, but the actual label in `LayerTogglesSlot.tsx` is "Climate". `page.getByLabel('Weather')` didn't match any element.
- **Fix:** Corrected the ALL_LAYERS constant to use "Climate" matching the actual UI.
- **Files modified:** `scripts/capture-hero.ts`
- **Verification:** All 6 layers toggle on successfully in subsequent runs.
- **Committed in:** `8f3ce82` (Task 1b commit)

**6. [Rule 1 — Bug] Screenshot capture order bleeds threat-cluster selection into search modal**

- **Found during:** Task 1 (hero GIF capture, screenshot sequence)
- **Issue:** Initial capture order was `detail-panel.png → search-modal.png`. detail-panel.png required clicking a threat cluster, which sets selectedCluster in uiStore. The cluster remained selected when search-modal.png was captured, making the search modal overlay a partially-dimmed map with a threat cluster highlight — muddied visual.
- **Fix:** Reordered so `search-modal.png` runs before `detail-panel.png`, from a clean map state.
- **Files modified:** `scripts/capture-hero.ts`
- **Verification:** Both screenshots now show their intended UI state without crosstalk.
- **Committed in:** `8f3ce82` (Task 1b commit)
