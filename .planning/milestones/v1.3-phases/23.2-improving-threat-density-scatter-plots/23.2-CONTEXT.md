# Phase 23.2: Improving Threat Density Scatter Plots - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Improve the visual appearance and interaction behavior of threat cluster scatter plot circles on the map. Covers gradient rendering, dual-dimension encoding (color=weight, radius=spread), z-order transitions, hover effects, and enriched detail panel content with geographic context. Does not add new data sources, change the clustering algorithm, or modify the threat weight formula.

</domain>

<decisions>
## Implementation Decisions

### Circle Rendering — Radial Gradient

- Replace solid-fill ScatterplotLayer with custom shader extension that renders radial gradients
- Center of each circle is full-opacity thermal color, fading to transparent at the edge (soft feathered edge)
- Additive blending via GL blend parameters (`SRC_ALPHA, ONE`) — overlapping circles blend and intensify
- Static circles only — no pulse or glow animation (consistent with Phase 23 decision)
- Custom GLSL fragment shader on ScatterplotLayer extension (~20 lines of shader code)

### Color Palette — Simplified Thermal

- Reduce from 8-stop FLIR Ironbow to 4-5 stop palette: deep purple → magenta → orange → bright red
- Color encodes threat weight (compound severity score from `computeThreatWeight`)
- P90 normalization retained from Phase 23

### Dual-Dimension Encoding

- **Radius** encodes geographic spread — how many grid cells the cluster spans (bounding box size)
- **Color** encodes threat weight — how dangerous the cluster is (severity score)
- These are decoupled: a large but low-severity cluster renders as a big purple circle; a small but high-severity cluster renders as a small bright-red circle

### Size Range — Pixel-Based

- Fixed pixel size, not meter-based (consistent with Phase 23 static sizing decision)
- Single-cell clusters: ~10-15px minimum
- Largest multi-cell clusters: ~100px+ maximum
- Size scales linearly with cluster cell count / bounding box diagonal

### Z-Order & Zoom Behavior

- Below zoom 9: clusters render ON TOP of event entity markers (clusters are primary viz at regional view)
- Above zoom 9: clusters drop BEHIND event entity markers (events take visual priority at city-level zoom)
- Cluster opacity stays constant at all zoom levels — no fade-out at high zoom
- Picking: deck.gl picks whichever geometry is closest to cursor, regardless of z-order

### Hover Interaction

- Tooltip content unchanged from Phase 23 (event count, relative time, dominant type, fatalities, hostility, mentions)
- Hovered cluster brightens/intensifies (opacity boost or color shift toward hot end)
- Non-hovered clusters dim to ~40% opacity when one is hovered (matches existing entity dimming pattern)

### Detail Panel Enrichment

- **Event type breakdown** — mini horizontal bars showing count per type (e.g., 12 Airstrikes, 5 Ground Combat), sorted by count descending
- **Geographic context** — two-tier lookup:
  1. First: check if any key site (from siteStore) falls within the cluster's bounding box → "Near [site name] (+N more sites)" format
  2. Fallback: reverse geocode cluster centroid via Nominatim (OpenStreetMap) → city/country name
- **Nominatim caching** — server-side reverse geocode with Redis cache (long TTL — geographic names don't change)
- **Event sort order** — events in scrollable list sorted by threat weight (highest severity first), not insertion order
- **Scroll position** — resets to top when re-opening cluster panel (no state preservation)
- **Existing fields kept** — event cards still show type, time, location, fatalities (no weight score added to cards)

### Claude's Discretion

- Exact GLSL shader code for radial gradient + alpha falloff curve
- Specific 4-5 hex color values for simplified thermal palette
- Pixel size interpolation function (linear vs sqrt vs log for cell-count-to-pixels mapping)
- Nominatim API endpoint selection and Redis key naming for geocode cache
- GL blend parameter configuration details
- How to implement the zoom-9 z-order crossover (layer ordering vs beforeId)

</decisions>

<specifics>
## Specific Ideas

- Gradient circles should feel like radar blips or thermal hotspots — hot center bleeding into the dark map
- Additive blending where circles overlap creates natural "super-hot" zones without explicit merge logic
- Size communicating spread and color communicating severity gives an instant two-axis read: "big purple = widespread but mild" vs "small red = concentrated and lethal"
- Geographic context priority: site name from siteStore first (no API call needed), Nominatim only as fallback for areas far from known infrastructure

</specifics>
