    1. In createApp():
       - Remove `const config = loadConfig();` (line 11)
       - For CORS origin, read `process.env.CORS_ORIGIN ?? 'http://localhost:5173'` directly instead of config.corsOrigin. This is the ONLY config value needed at app construction time, and it has a safe default.

    2. In the isMainModule block:
       - Remove `const config = loadConfig();` (line 40)
       - For the port, read `Number(process.env.PORT ?? 3001)` directly instead of config.port
       - Guard connectAISStream() behind an env var check: only call it if `process.env.AISSTREAM_API_KEY` is set. Log a warning if skipped: `console.log('[server] AISSTREAM_API_KEY not set, skipping AIS WebSocket connection')`

    3. Remove the `import { loadConfig } from './config.js'` since it is no longer used in index.ts. The adapters already import the lazy Proxy `config` and access credentials only inside their async functions — no changes needed to adapters.

    4. In server/config.ts — no functional changes needed. The Proxy-based `config` export and `loadConfig()` function stay as-is for use by adapters and tests. The only change: add an exported helper function `getServerConfig()` that returns ONLY port and corsOrigin from process.env without validating API credentials:
       ```typescript
       export function getServerConfig(): { port: number; corsOrigin: string } {

           port: Number(process.env.PORT ?? 3001),

         };
       }
       ```

       Actually — on reflection, since index.ts only needs port and corsOrigin, and both are simple one-liners reading process.env, using inline process.env reads in index.ts is simpler and avoids an unnecessary abstraction.     Result: Server boots, mounts routes, listens on port — all without requiring any API credentials. Credentials are only validated when an adapter function is called (e.g., when a user hits /api/flights).
